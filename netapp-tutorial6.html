SECTION 01

Disk Ownership & Node-Level Spare Pools

Every physical disk in an ONTAP cluster is owned by exactly one node. Spare disks are node-specific — to expand an aggregate on stor-node-01, every disk added must belong to that node's spare pool. Cross-node disk assignment is not permitted without first transferring ownership.

Figure 1 — Disk ownership is strictly per-node. Cross-node disk assignment to aggregates is not permitted.
ONTAP CLI — Reassign Disk Ownership
# Step 1: Remove ownership from current node disk remove-ownership -disk 2.0,2.1,2.2,2.3,2.4,2.5 # Step 2: Assign to target node disk assign -disk-list 2.0,2.1,2.2,2.3,2.4,2.5 -owner stor-node-01 # Verify spare pool update storage aggregate show-spare-disks -original-owner stor-node-01
⚠️
Warning
Removing disk ownership temporarily reduces spare capacity. Always verify the remaining spare count meets your protection threshold before proceeding. Never leave a node with zero spares.

SECTION 02

Expanding an Aggregate — Three Strategies

When an aggregate approaches capacity, you have three expansion paths. Choosing the right one depends on your current RAID group layout and available spare inventory.

Strategy 1 — New RAID Group
storage aggregate add-disks \ -aggregate aggr_data01 \ -diskcount 6 \ -disktype SSD \ -raidgroup new storage aggregate show-status -aggregate aggr_data01

Strategy 2 — Add to an Existing RAID Group

When spare inventory is critically low, expand the max-raidgroup-size and insert disks into existing groups as data disks (no parity overhead). This is an emergency measure — the size increase is irreversible.

🚨
Critical — Irreversible Action
Increasing the max-raidgroup-size cannot be undone. Disks also cannot be removed from an aggregate once added — the only path back is destroying the entire aggregate.
Strategy 2 — Expand Existing RG
# Step 1: Increase RAID group size (irreversible) storage aggregate modify -aggregate aggr_data01 -maxraidsize 7 # Step 2: Add one disk to rg0 as a data disk storage aggregate add-disks \ -aggregate aggr_data01 \ -diskcount 1 \ -raidgroup rg0

Strategy 3 — Auto-Distribute (all)

The raidgroup: all option lets ONTAP identify available slots across all RAID groups and fill them automatically. Any disks exceeding available slots are returned to the spare pool.

Strategy 3 — Auto-Distribute
storage aggregate add-disks \ -aggregate aggr_data01 \ -diskcount 4 \ -raidgroup all # Warning: 4 disks specified, but only 2 slots available. # 2 disks will be added; 2 disks returned to spare pool.

RAID Group Sizing Guidelines
For production aggregates, a RAID group size between 17 and 23 disks gives the best balance of performance and protection. Under 17 wastes parity overhead; above 23 increases rebuild time risk.

Reallocation — Block-Level Defragmentation

Over time, block allocation within an aggregate becomes fragmented. Reallocation reorganizes blocks so used blocks are contiguous, improving sequential I/O performance.

Figure 3 — Reallocation consolidates fragmented blocks, improving sequential I/O performance similar to disk defragmentation.

Relocation — Moving an Aggregate Between Nodes

Within an HA pair, you can non-disruptively move an aggregate from one node to its partner using aggr relocation. Disk ownership transfers automatically. Clients experience zero disruption.

Figure 4 — Aggregate relocation is non-disruptive. Clients experience no interruption while aggregates move between HA pair nodes. Disk ownership transfers automatically.
ℹ️
Relocation Boundaries
Aggregate relocation only works between nodes in the same HA pair. You cannot relocate an aggregate to a node in a different HA pair. The SFO (Storage Failover) mechanism handles this during planned and unplanned failover scenarios.
ONTAP CLI · Relocation
storage aggregate relocation start \ -node stor-node-01 \ -destination stor-node-02 \ -aggregate-list aggr_data01 storage aggregate relocation show

SECTION 04

RAID Scrub & Media Scrub

Disk drives accumulate bad sectors over time. ONTAP addresses this through two complementary background scanning processes.

Figure 5 — RAID Scrub scans each disk track, identifies unreadable sectors (✗), reconstructs data from parity stripes, and writes recovered data back to healthy sectors (✓).

ONTAP CLI · Scrub Management
storage aggregate scrub -aggregate aggr_data01 -action start storage aggregate scrub -aggregate aggr_data01 -action status storage aggregate scrub -aggregate aggr_data01 -action suspend # Schedule scrub every Friday at 23:00; set low performance impact storage raid-options modify -node * -name raid.scrub.schedule -value "Fri@23" storage raid-options modify -node * -name raid.scrub.perf_impact -value low

Pre-Fail: Proactive Disk Replacement

When ONTAP detects a rising bad-sector count, it triggers pre-fail — copying data from the degrading disk to a spare before marking it failed. This avoids a full RAID reconstruction.

SECTION 06

Mirror Aggregates — Concept & Architecture

A mirror aggregate synchronously duplicates all data across two independent sets of disks called plexes. If one disk shelf becomes unavailable, the aggregate continues from its surviving plex — zero data loss, zero downtime. Mirror aggregates are primarily used in MetroCluster configurations where HA pair nodes sit in separate data centers.

Figure 7 — MetroCluster architecture: mirror aggregate spans two independent disk shelves across geographically separated data centers.

Plexes and Spare Pools

Normal aggregates have only Plex 0. Mirror aggregates add Plex 1 — an exact live duplicate. Disks for Pool 0 come from Shelf A; Pool 1 from Shelf B, ensuring a single shelf failure only takes down one plex.

ONTAP CLI · Mirror Aggregate Creation
# Assign Shelf A → Pool 0, Shelf B → Pool 1 disk assign -disk-list 1.0,1.1,...,1.11 -owner stor-node-01 -pool 0 disk assign -disk-list 2.0,2.1,...,2.11 -owner stor-node-01 -pool 1 # Create mirror aggregate storage aggregate create \ -aggregate aggr_mirror01 \ -node stor-node-01 \ -raidtype raid_dp \ -maxraidsize 8 \ -disklist 1.0,1.1,1.2,1.3,1.4,1.5,1.6,1.7,1.8,1.9 \ -mirror-disklist 2.0,2.1,2.2,2.3,2.4,2.5,2.6,2.7,2.8,2.9
⚠️
Mirror Aggregate — Space Cost
Mirroring doubles the disk requirement. Expect ~40–45% usable storage from raw disk count due to combined mirroring and RAID overhead. Only Plex 0's data-disk capacity counts as usable.

SECTION 08

SVM / vServer — The Heart of ONTAP Cluster-Mode

A Storage Virtual Machine (SVM) — also called a vServer — is ONTAP's logical data-access layer. It acts as a virtual storage appliance within the cluster, presenting storage to clients while completely hiding the physical topology. Clients connect to SVMs — never directly to nodes.

💡
Multi-Tenant SVM Design
When hosting multiple tenants or segregating SAN from NAS workloads, create separate SVMs rather than mixing protocols into one. This provides security isolation, independent protocol configuration, and cleaner operational boundaries. Add IP Spaces for complete network-level isolation between tenant SVMs.

SECTION 10

Interview Q&A — Scenario-Based Questions

Q1 › You need to expand an aggregate urgently but only have 3 spare disks — not enough for a new RAID group. How do you proceed?
Increase maxraidsize using storage aggregate modify -maxraidsize to allow one extra disk per existing RAID group. Then add spares individually to specific RAID groups using -raidgroup rg0 etc. The added disks become data disks — no parity overhead. Critical: this change is irreversible, so document it and order replacement disks immediately.
Q2 › An admin added 4 disks using -raidgroup all but ONTAP only added 2. Why?
The raidgroup: all option scans all RAID groups, identifies available slots based on maxraidsize, and fills only those slots. Only 2 slots were available, so ONTAP added 2 disks and returned the remaining 2 to the spare pool. No harm done.
Q3 › What is the difference between aggregate reallocation and aggregate relocation?
Reallocation is block-level defragmentation within a single aggregate — it reorganizes used and free blocks for better I/O performance. It does not move the aggregate. Relocation physically moves the aggregate from one node to its HA partner — disk ownership transfers automatically and the operation is non-disruptive to clients.
Q4 › Why does a mirror aggregate provide only ~40% usable capacity from raw disks?
Three overhead layers apply: (1) Disk-level: each drive provides ~90% of rated capacity — the rest is reserved for firmware and bad-block remapping. (2) RAID overhead: RAID-DP uses 2 disks per RAID group for parity and double-parity. (3) Mirroring overhead: the entire RAID structure is duplicated in Plex 1, so you consume twice the disk count for the same data. Only Plex 0's data-disk capacity is usable.
Q5 › What are the four SVM types in ONTAP, and which one do clients interact with?
Admin SVM: cluster management — auto-created. Node SVM: per-node management — auto-created. Data SVM: client-facing — admin-created; hosts volumes, LUNs, LIFs, and protocols. System SVM: internal ONTAP operations — auto-created, diagnostic mode only. Clients exclusively interact with Data SVMs through their configured LIF IP addresses.
Q6 › In a 4-node cluster, can a Data SVM provision a volume from node-3's aggregate and serve it via a node-1 LIF?
Yes — this is a key advantage of Cluster-Mode. A Data SVM has visibility into all aggregates across all nodes. A volume on aggr_node03 can be served through any LIF on any node. ONTAP routes the I/O internally. The client never knows which physical node owns the storage, enabling flexible capacity planning and non-disruptive volume mobility.
Q7 › What is RAID Scrub and how does it differ from Media Scrub?
RAID Scrub: manually triggered or scheduled; scans all disks in aggregates for hardware and media errors; reconstructs data from bad sectors using parity; has configurable performance impact; should run during off-peak hours. Media Scrub: always-on background process at low intensity; covers both active aggregate disks and spare drives; detects media errors only. They are complementary — media scrub for continuous monitoring, RAID scrub for periodic deep validation.
