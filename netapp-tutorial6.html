SECTION 01

Disk Ownership & Node-Level Spare Pools

In a ONTAP cluster, every physical disk is owned by exactly one node. This ownership determines which node's aggregates can consume that disk. Understanding this boundary is critical before any expansion activity.

Why Ownership Matters for Expansion

Spare disks are node-specific. If you intend to expand an aggregate on stor-node-01, every disk added must belong to stor-node-01's spare pool. Disks assigned to stor-node-02 cannot be directly added — ownership must be transferred first.

Reassigning Disk Ownership

To move disks between nodes, you first remove ownership, then reassign to the target node. This is a two-step process:

ONTAP CLI
# Step 1: Remove ownership from current node (multiple disks with comma-separation) disk remove-ownership -disk 2.0,2.1,2.2,2.3,2.4,2.5 # Verify disks no longer appear in stor-node-02 spare pool storage disk show -owner stor-node-02 # Step 2: Assign disks to target node (stor-node-01) disk assign -disk-list 2.0,2.1,2.2,2.3,2.4,2.5 -owner stor-node-01 # Verify spare pool update storage aggregate show-spare-disks -original-owner stor-node-01
⚠️
Warning — Production Impact
Removing disk ownership from a node temporarily reduces its spare capacity. Always verify the remaining spare count meets your protection threshold before proceeding. Never leave a node with zero spares.

SECTION 02

Expanding an Aggregate — Three Strategies

When an aggregate approaches capacity, you have three distinct expansion paths. Each is suited to a different operational situation. Choosing the right strategy requires understanding your current RAID group layout and available spare inventory.

Strategy 1 — Create a New RAID Group

When your spare pool is healthy and you have at least as many spares as your configured RAID group size, creating a new RAID group is the cleanest approach. This maintains consistent group sizing and preserves NetApp best practices.

✅
Best Practice
Maintain the same number of disks across all RAID groups within a single aggregate. Uneven RAID groups can create performance imbalance. If your existing groups have 8 disks each, add exactly 8 disks to the new group.

ONTAP CLI · Strategy 1
# Add 6 disks to aggr_data01 in a brand-new RAID group storage aggregate add-disks \ -aggregate aggr_data01 \ -diskcount 6 \ -disktype SSD \ -raidgroup new # Verify new RAID group was created storage aggregate show-status -aggregate aggr_data01

Strategy 2 — Add Disks to an Existing RAID Group

When spare inventory is critically low — not enough to form a complete new RAID group — you can expand the max-raidgroup-size setting and insert individual disks into existing groups. This avoids consuming two extra disks on parity overhead for a new group, buying time until fresh hardware arrives.

🚨
Critical — Irreversible Action
Increasing the max-raidgroup-size is a one-way operation. Once increased, you cannot reduce it. Additionally, disks cannot be removed from an aggregate once added — the only path back is destroying the entire aggregate. Plan carefully before executing.
ONTAP CLI · Strategy 2
# Step 1: Increase RAID group size from 6 to 7 (irreversible) storage aggregate modify \ -aggregate aggr_data01 \ -maxraidsize 7 # Step 2: Add one disk to RAID group rg0 specifically storage aggregate add-disks \ -aggregate aggr_data01 \ -diskcount 1 \ -raidgroup rg0 # The new disk is added as a DATA disk (no additional parity cost) storage aggregate show-status -aggregate aggr_data01

Strategy 3 — Intelligent Auto-Distribution (all)

The raidgroup: all option instructs ONTAP to analyze the aggregate layout, identify available slots across all RAID groups based on the max-raidgroup-size setting, and automatically fill them. Any disks you provide that exceed available slots are returned to the spare pool — no waste, no manual slot calculation.

ONTAP CLI · Strategy 3
# Provide more disks than slots — ONTAP places only what fits storage aggregate add-disks \ -aggregate aggr_data01 \ -diskcount 4 \ -raidgroup all # ONTAP output (example): # Warning: 4 disks specified, but only 2 slots available. # 2 disks will be added; 2 disks returned to spare pool. # Result: balanced RAID groups, no leftover slots, spare pool intact storage aggregate show-status -aggregate aggr_data01

SECTION 03

Auto-Provisioning Feature

ONTAP provides an aggr auto-provision command that attempts to create an aggregate automatically by following NetApp's recommended sizing guidelines. It evaluates your spare pool, picks an optimal RAID group size, reserves the minimum required spares, and creates the aggregate — including naming.

🚨
Warning — Use With Caution
Auto-provisioning may not honor your organization's specific capacity planning requirements or RAID group size conventions. In production environments, always perform manual analysis: calculate usable space, verify RAID group sizing, and validate against your storage policy before creating any aggregate. Do not rely on automation for business-critical storage builds.
ONTAP CLI
# Auto-provision creates aggregate with NetApp defaults storage aggregate auto-provision -node stor-node-01 # Review what was created storage aggregate show-status
💡
RAID Group Sizing Guidelines
For production aggregates, a RAID group size between 17 and 23 disks provides the best balance of performance and protection. Under 17 disks wastes too much space on parity; above 23 increases rebuild time risk. Within this range, tailor to your specific drive count and capacity needs.

Reallocation — Defragmentation for Storage

Over time, as volumes are created and deleted, the block allocation within an aggregate becomes fragmented. Free blocks and used blocks become interleaved, which can degrade I/O performance. Reallocation is ONTAP's answer — it reorganizes blocks so that all used blocks are contiguous and all free blocks form a single large pool.

Relocation — Moving an Aggregate Between Nodes

Within an HA pair, you can non-disruptively move an aggregate from one node to its partner using aggr relocation. Disk ownership transfers automatically. This is particularly useful during planned maintenance — when a node is being rebooted, ONTAP automatically relocates its aggregates to the HA partner.

Figure 4 — Aggregate relocation is non-disruptive. Clients experience no interruption while aggregates move between HA pair nodes. Disk ownership transfers automatically.
ℹ️
Relocation Boundaries
Aggregate relocation only works between nodes in the same HA pair. You cannot relocate an aggregate to a node in a different HA pair within the cluster. The SFO (Storage Failover) mechanism handles this during planned and unplanned failover scenarios.
ONTAP CLI · Relocation
# Relocate aggregate from stor-node-01 to stor-node-02 storage aggregate relocation start \ -node stor-node-01 \ -destination stor-node-02 \ -aggregate-list aggr_data01 # Monitor relocation progress storage aggregate relocation show

SECTION 05

RAID Scrub & Media Scrub

Disk drives accumulate bad sectors over time due to media degradation or hardware wear. A bad sector is a region on the disk platter that becomes unreadable, meaning data stored there can no longer be retrieved directly. ONTAP addresses this through two complementary background scanning processes: RAID Scrub and Media Scrub.

Figure 5 — RAID Scrub scans each disk track, identifies unreadable sectors (✗), reconstructs data from parity stripes, and writes recovered data back to healthy sectors (✓).

Scheduling and Configuring RAID Scrub

ONTAP CLI · Scrub Management
# Start scrub immediately on a specific aggregate storage aggregate scrub -aggregate aggr_data01 -action start # Check scrub progress (% complete) storage aggregate scrub -aggregate aggr_data01 -action status # Suspend if impacting business-hours performance storage aggregate scrub -aggregate aggr_data01 -action suspend # Schedule scrub every Friday at 23:00 on all nodes storage raid-options modify \ -node * \ -name raid.scrub.schedule \ -value "Fri@23" # Set performance impact to LOW (recommended for scheduled runs) storage raid-options modify \ -node * \ -name raid.scrub.perf_impact \ -value low # Set timeout to 24 hours to auto-suspend before business hours start storage raid-options modify \ -node * \ -name raid.scrub.duration \ -value 24
✅
Best Practice — Scrub Configuration
Always schedule RAID scrubs during off-peak windows (weekends or overnight). Set performance impact to low and configure a timeout so that scrub operations automatically suspend before business hours begin. This prevents latency complaints from clients during peak activity.

Pre-Fail: Proactive Disk Replacement

When ONTAP detects a rising bad-sector count on a drive, it can trigger pre-fail — a graceful, data-safe replacement sequence. Rather than waiting for an outright disk failure, ONTAP copies data from the degrading disk to a spare drive before marking it as failed. This significantly reduces RAID reconstruction time and risk.

Key Space Facts

SECTION 07

Mirror Aggregates — Concept & Architecture

A mirror aggregate is a special aggregate type where all data is synchronously duplicated across two independent sets of disks called plexes. If one disk shelf or pool of disks becomes unavailable, the aggregate continues operating from its surviving plex — with zero data loss and zero downtime.

When Are Mirror Aggregates Used?

Mirror aggregates are primarily associated with MetroCluster configurations — where HA pair nodes reside in physically separate data centers (up to 5 km apart). If one site suffers a power failure or hardware catastrophe, the partner site's node takes over using the surviving plex of the mirror aggregate.

Understanding Plexes and Spare Pools

Every aggregate has at least one plex. A plex is a logical wrapper over a set of RAID groups — think of it as a complete, self-contained copy of the aggregate's RAID structure. Normal aggregates have only Plex 0. Mirror aggregates add a second plex (e.g., Plex 1) that contains an exact, live duplicate.

Spare Pool Architecture for Mirror Aggregates

ONTAP's spare pool is sub-divided into Pool 0 and Pool 1. By default, all disks are assigned to Pool 0. For mirror aggregates, you intentionally split disk shelves — all disks from Shelf A go to Pool 0, all disks from Shelf B go to Pool 1. This physical separation ensures that a single shelf failure only takes down one plex, not both.

ONTAP CLI · Mirror Aggregate Creation
# Assign Shelf A disks to Pool 0 (default) disk assign -disk-list 1.0,1.1,1.2,1.3,1.4,1.5,1.6,1.7,1.8,1.9,1.10,1.11 \ -owner stor-node-01 -pool 0 # Assign Shelf B disks to Pool 1 disk assign -disk-list 2.0,2.1,2.2,2.3,2.4,2.5,2.6,2.7,2.8,2.9,2.10,2.11 \ -owner stor-node-01 -pool 1 # Verify both pools are populated storage aggregate show-spare-disks -original-owner stor-node-01 # Create mirror aggregate using disks from both pools storage aggregate create \ -aggregate aggr_mirror01 \ -node stor-node-01 \ -raidtype raid_dp \ -maxraidsize 8 \ -disklist 1.0,1.1,1.2,1.3,1.4,1.5,1.6,1.7,1.8,1.9 \ -mirror-disklist 2.0,2.1,2.2,2.3,2.4,2.5,2.6,2.7,2.8,2.9
⚠️
Mirror Aggregate — Space Cost
Mirroring doubles the disk requirement. Every byte written is stored in both plexes simultaneously. The usable capacity is determined by Plex 0 alone (parity-adjusted), while Plex 1 holds the duplicate. Plan your capacity accordingly — expect ~40–45% usable storage from your raw disk count due to combined mirroring and RAID overhead.

SECTION 08

Mirror Aggregate Operations

Once a mirror aggregate is in place, you have several lifecycle operations available — converting between mirrored and non-mirrored states, expanding capacity, and verifying plex consistency.

Converting Mirror → Normal Aggregate

If the data hosted on a mirror aggregate no longer requires the highest availability tier, you can remove the mirror plex. This releases all disks in the secondary plex back to the spare pool.

ONTAP CLI
# Remove the mirror plex (demirror) storage aggregate plex delete \ -aggregate aggr_mirror01 \ -plex plex1 # Aggregate now shows as non-mirrored; plex1 disks return to spare pool storage aggregate show -aggregate aggr_mirror01

Converting Normal → Mirror Aggregate

You can add a mirror plex to an existing normal aggregate at any time — no downtime required. ONTAP creates the new plex, then synchronizes all existing data in the background.

ONTAP CLI
# Add mirror plex to existing aggregate (same disk count as original) storage aggregate mirror \ -aggregate aggr_data01 \ -mirror-disklist 2.0,2.1,2.2,2.3,2.4,2.5,2.6,2.7,2.8,2.9 # Monitor resync progress (aggregate shows "mirror degraded" until sync completes) storage aggregate show-status -aggregate aggr_data01

Verifying Plex Data Consistency

The aggr verify operation confirms that both plexes contain identical data — essential after a resync or hardware replacement. It compares block-by-block and reports any discrepancies.

ONTAP CLI
# Start plex verification storage aggregate verify -aggregate aggr_mirror01 -action start # Check verification status storage aggregate verify -aggregate aggr_mirror01 -action status
🎯
Resynchronization Priority
If a plex goes offline and comes back online, ONTAP triggers a resync automatically. You can adjust the priority (low, medium, high) to control how aggressively it consumes controller resources. In production, always keep this at low to avoid performance impact during business hours.

SECTION 09

SVM / vServer — The Heart of ONTAP Cluster-Mode

A Storage Virtual Machine (SVM) — also called a vServer — is ONTAP's foundational abstraction layer for data access. It acts as a logical storage appliance within the cluster, presenting storage to clients while completely hiding the physical cluster topology.

Figure 8 — SVM acts as the data-access gateway. Clients see only the SVM's LIFs and volumes — never the physical cluster nodes or aggregates.

The SVM architecture eliminates a fundamental limitation of legacy storage operating modes: in earlier designs, storage was presented directly from a controller — meaning clients were tied to specific hardware. With SVMs, volumes can reside on any node's aggregate in the cluster, and the SVM transparently handles routing. Client access is not disrupted even when volumes are moved between nodes.

The Four SVM Types

Represents each physical controller (node) in the cluster. Used for node-level management. One created per node during cluster setup. Handles AutoSupport (node-to-NetApp communication).

📦

The most important type — created by the storage administrator. This is where volumes, LUNs, LIFs, and protocols are configured. All client data access flows through Data SVMs.

Hidden from standard view — accessible only in diagnostic mode. Manages cluster-internal communication, inter-node traffic, and background ONTAP services. Entirely managed by ONTAP itself.

Multi-Tenancy and SVM Segregation

One of the most powerful SVM use cases is multi-tenancy — hosting multiple independent organizations or application tiers on the same physical cluster while maintaining strict isolation between them. Each tenant gets its own SVM with dedicated protocols, IP addresses, and access controls.

💡
Multi-Tenant SVM Design
When hosting multiple tenants or segregating SAN from NAS workloads, create separate SVMs rather than mixing everything into one. This provides security isolation, independent protocol configuration, and cleaner operational boundaries. Add IP Spaces for complete network-level isolation between tenant SVMs.

SVM Protocol Flexibility

A single Data SVM can serve multiple protocols simultaneously — NFS, SMB/CIFS, iSCSI, and FCP can all be configured on the same SVM. Or you can create dedicated SVMs per protocol for environments requiring strict separation.

SECTION 11

Interview Q&A — Scenario-Based Questions

These questions are modeled after real storage engineer interview scenarios covering the concepts from this tutorial.

Q1 › You need to expand an aggregate urgently but only have 3 spare disks — not enough to form a new RAID group. How do you proceed without violating data integrity?
Increase the aggregate's maxraidsize setting using storage aggregate modify -maxraidsize to allow one additional disk per existing RAID group. Then add the available spares individually to specific RAID groups using -raidgroup rg0, rg1, etc. The added disks become data disks (no parity overhead). Critical note: this max-raidgroup-size change is irreversible, so document the change and immediately order replacement disks to restore a healthy spare pool.
Q2 › A storage administrator added 4 disks to an aggregate using the -raidgroup all option, but the system only added 2 disks. Why?
The raidgroup: all option is intelligent — it scans all RAID groups within the aggregate, identifies how many available slots exist based on the current maxraidsize setting, and fills only those slots. In this case, only 2 slots were available across all existing RAID groups, so ONTAP added 2 disks and returned the remaining 2 to the spare pool. No data or configuration was harmed.
Q3 › What is the difference between aggregate reallocation and aggregate relocation? They sound similar.
Reallocation is a block-level defragmentation process that runs on a single aggregate. It reorganizes the used and free block distribution within the aggregate to improve I/O performance — conceptually similar to disk defragmentation in a traditional OS. It does not move the aggregate. Relocation, by contrast, physically moves the aggregate from one node to its HA partner within the same HA pair. Disk ownership transfers automatically, and the operation is non-disruptive to clients.
Q4 › Why does a mirror aggregate provide only ~40% usable capacity from raw disk? Explain the overhead layers.
Three layers consume capacity: (1) Disk-level: each physical disk provides only ~90% of its rated capacity, with the remainder reserved for firmware and bad-block remapping. (2) RAID overhead: in RAID-DP, 2 disks per RAID group are used for parity and double-parity — not for data. (3) Mirroring overhead: the entire RAID structure is duplicated in Plex 1, meaning you effectively consume twice the disk count to store the same data. Only Plex 0's data-disk capacity is usable. The 10% aggregate reserve adds a final layer of overhead.
Q5 › What are the four types of SVMs in ONTAP, and which one do clients interact with?
Admin SVM: represents the cluster for administrative management — created automatically. Node SVM: represents each physical node — one per node, created automatically. Data SVM: the client-facing SVM where volumes, LUNs, LIFs, and protocols live — created manually by the administrator. System SVM: manages internal cluster operations (visible only in diagnostic mode) — fully managed by ONTAP. Clients always interact exclusively with Data SVMs through their configured LIF IP addresses.
Q6 › In a 4-node ONTAP cluster, can a Data SVM provision a volume from a node-3 aggregate and serve it to a client connecting through a node-1 network interface?
Yes — this is one of the key architectural advantages of Cluster-Mode over 7-Mode. A Data SVM has visibility into all aggregates across all cluster nodes. A volume can be created on aggr_node03 and associated with the SVM. The client connects through any LIF on any node — ONTAP routes the I/O internally. The client never needs to know which physical node owns the storage. This abstraction enables flexible capacity planning and non-disruptive volume mobility.
Q7 › What is RAID Scrub and how does it differ from Media Scrub? When would you run each?
RAID Scrub is a manually-triggered or scheduled background process that scans all disks within aggregates (plexes and RAID groups) for both hardware and media errors. It reconstructs data from bad sectors using RAID parity and writes recovered data to healthy sectors. It has configurable performance impact (low/medium/high) and should be scheduled during off-peak hours. Media Scrub runs continuously by default at low intensity and covers both active aggregate disks and spare drives — but only checks for media errors (not hardware errors). It requires no scheduling and has minimal performance impact. Both are complementary: media scrub for constant background health monitoring, RAID scrub for periodic deep validation.
