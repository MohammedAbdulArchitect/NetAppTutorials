SECTION 01

NFS — Network File System

In the NAS (Network Attached Storage) world, ONTAP supports two primary protocols for sharing storage over a network. SMB/CIFS caters to Windows-based systems, while NFS (Network File System) is the standard for Linux, Unix, and other POSIX-compliant operating systems. NFS allows remote file access over a standard IP network, making storage appear as locally mounted directories to client servers.

📌 NFS Protocol Versions

Multiple NFS versions are available in ONTAP. Choosing the right version depends on your client operating system capabilities and organizational requirements.

VersionStatusKey FeaturesUsage in Practice
NFSv2LegacyBasic file access, UDP onlyRarely used, considered obsolete
NFSv3DominantTCP + UDP, async writes, large file support~95% of enterprise deployments
NFSv4ModernACLs, stateful protocol, improved securityPOC/testing, growing adoption
NFSv4.1Modern+pNFS, session trunking, delegationSelect high-performance environments
⚠️
Version Selection Guidance
NFSv3 remains the industry standard due to its proven stability. NFSv4/4.1 introduces ACL-level permissions (multiple owners/groups per resource) which NFSv3 lacks, but the operational complexity means most organizations stick with v3 unless there's a specific business requirement. Always align version selection with the capabilities of your client OS — RHEL 7.x pairs well with NFSv3; RHEL 8+ supports both v3 and v4.

SECTION 02

Export Policy Architecture

Export policies are ONTAP's mechanism for controlling which NFS clients can access which volumes, and what level of access they receive. Think of export policies as gatekeepers — each volume has exactly one export policy assigned to it, and that policy contains one or more rules defining allowed clients and access types.

Figure 4 — Layered export policy model: default policy at junction, custom policies per volume
🏡
Analogy: Colony Gate vs Individual Houses
The default export policy at the junction point is like the entry gate of a housing colony — all residents (servers) can enter the colony. But to access a specific house (volume), they need that house's own key (custom export policy with their IP). A server not listed in the volume's policy simply cannot enter that house, even though it passed the gate.
🔑 Authentication Mechanisms in Export Rules
MechanismDescriptionUse Case
sysUnix UID/GID based authenticationDefault for all Linux/Unix systems — use this in 99% of cases
anyAccepts all authentication methodsFlexible mixed environments
krb5Kerberos v5 — basicHigh-security environments requiring Kerberos
krb5iKerberos v5 with data integrityKerberos + checksum verification
krb5pKerberos v5 with full privacyKerberos + full encryption
ntlmWindows NTLM authenticationWindows-initiated NFS access
none / neverDeny accessExplicitly block certain access levels
⚠️
Superuser is Mandatory
Every export policy rule that should allow mounting must include superuser (typically set to sys). Without superuser access, Linux clients will be unable to mount the volume, even if RO/RW access is granted. A rule combination of -superuser sys -rw sys -ro sys is the most common configuration.

SECTION 03

Creating Export Policies & Rules

Let's walk through the complete workflow of creating an export policy, defining rules, and associating the policy with a volume. We'll use a realistic lab scenario with two Linux application servers requiring different access levels.

Lab Scenario

🧪 Lab Setup
  • SVM Name: svm-prod
  • Linux Server A: app-server-01 — IP: 10.1.1.21 — needs Read-Write access
  • Linux Server B: app-server-02 — IP: 10.1.1.22 — needs Read-Only access
  • Volume: vol-app01 (10 GB, junction path: /vol-app01)
  • Protocol: NFSv3

Step 1 — Create the Default Export Policy Rule (Junction Point)

ONTAP CLI — Default Policy (Junction Point)
# Check existing export policies on SVM cluster01::> vserver export-policy show -vserver svm-prod # Check rules in the default policy (should be empty initially) cluster01::> vserver export-policy rule show -vserver svm-prod -policyname default # Create a default rule — allows all IPs, Read-Only, no superuser # This acts as the gateway rule at the junction point cluster01::> vserver export-policy rule create \ -vserver svm-prod \ -policyname default \ -clientmatch 0.0.0.0/0 \ -rorule sys \ -rwrule never \ -superuser none \ -protocol nfs3 \ -anon 65534 \ -ruleindex 1

Step 2 — Create a Custom Export Policy for the Volume

ONTAP CLI — Custom Policy
# Create new export policy cluster01::> vserver export-policy create -vserver svm-prod -policyname app01_policy # Rule 1: app-server-01 gets Read-Write access cluster01::> vserver export-policy rule create \ -vserver svm-prod \ -policyname app01_policy \ -clientmatch 10.1.1.21 \ -rorule sys \ -rwrule sys \ -superuser sys \ -protocol nfs3 \ -anon 65534 \ -ruleindex 1 # Rule 2: app-server-02 gets Read-Only access cluster01::> vserver export-policy rule create \ -vserver svm-prod \ -policyname app01_policy \ -clientmatch 10.1.1.22 \ -rorule sys \ -rwrule never \ -superuser sys \ -protocol nfs3 \ -anon 65534 \ -ruleindex 2 # Verify the rules cluster01::> vserver export-policy rule show -vserver svm-prod -policyname app01_policy

Step 3 — Create Volume and Apply Export Policy

ONTAP CLI — Volume Create with Policy
# Create volume — apply export policy at creation time cluster01::> volume create \ -vserver svm-prod \ -volume vol_app01 \ -aggregate aggr_data_01 \ -size 50g \ -security-style unix \ -space-guarantee none \ -junction-path /vol_app01 \ -policy app01_policy # Or modify an existing volume to use a specific policy cluster01::> volume modify -vserver svm-prod -volume vol_app01 -policy app01_policy # Verify the policy applied to volumes cluster01::> volume show -vserver svm-prod -fields policy
💡
Reuse Policies Across Multiple Volumes
A single export policy can be applied to multiple volumes. If two volumes share the same access requirements — same list of servers, same access types — assign the same policy to both. No need to create duplicate policies. This keeps your configuration clean and maintainable.

SECTION 04

Mounting NFS Volumes on Linux

Once the export policy is correctly configured on the ONTAP side, Linux administrators can mount the storage volume. The mount path follows the format: SVM-DNS-or-LIF-IP:/junction-path. In production, this work is handled by the Linux/Unix team — as a storage engineer you provide them the mount path.

📂 Mount Path Formats

🔵 Using DNS Name (Recommended)
Linux Shell
# Create local mount point mkdir -p /mnt/app-data # Mount using SVM DNS name mount svm-prod.corp.local:/vol_app01 /mnt/app-data # Verify the mount df -h # List contents ls /mnt/app-data
🟡 Using LIF IP (Direct)
Linux Shell
# Find data LIF IPs on ONTAP cluster01::> network interface show \ -vserver svm-prod -role data # Mount using LIF IP directly mount 10.1.1.100:/vol_app01 /mnt/app-data # 10.1.1.100 = data LIF IP
⚖️ LIF IP vs DNS Name — Which to Use?
AspectLIF IPDNS Name
High AvailabilitySingle point of failureMultiple LIFs → HA
LIF FailoverConnection drops if LIF failsDNS resolves to surviving LIF
SimplicitySimple, no DNS neededRequires DNS configuration
Best ForTesting / troubleshootingProduction environments
🚨
Production Best Practice
Always use the DNS name (SVM name) for production NFS mounts. Multiple data LIFs are typically bound to the same DNS entry — if one LIF goes down, the DNS still resolves to the other LIF, keeping the mount active. Hardcoding a LIF IP creates an unnecessary single point of failure.

🔍 Auto-Discover Volumes with showmount

ONTAP includes a handy feature where, if enabled, Linux clients can query the SVM and see all volumes they have access to — without needing to know the exact junction path.

ONTAP CLI + Linux Shell
# Enable showmount on SVM (ONTAP side) cluster01::> vserver nfs modify -vserver svm-prod -showmount enabled # From Linux server — discover all accessible volumes showmount -e svm-prod.corp.local Export list for svm-prod.corp.local: /vol_app01 10.1.1.21,10.1.1.22 /vol_db01 10.1.1.31 /vol_backup01 10.1.1.25,10.1.1.26

SECTION 05

Managing Export Policy Rules

As your environment evolves, you'll frequently need to add new clients, remove decommissioned servers, modify access levels, or reorganize policies. ONTAP provides granular commands for all of these operations.

➕ Add / Remove Clients from an Existing Rule

ONTAP CLI — Client Management
# Add a new server to an existing rule (same access level) cluster01::> vserver export-policy rule add-clientmatch \ -vserver svm-prod \ -policyname app01_policy \ -ruleindex 1 \ -clientmatch 10.1.1.23 # Remove a specific server from a rule cluster01::> vserver export-policy rule remove-clientmatch \ -vserver svm-prod \ -policyname app01_policy \ -ruleindex 1 \ -clientmatch 10.1.1.23 # Modify an existing rule (e.g., change access level) cluster01::> vserver export-policy rule modify \ -vserver svm-prod \ -policyname app01_policy \ -ruleindex 2 \ -rwrule sys # Change the rule index number cluster01::> vserver export-policy rule setindex \ -vserver svm-prod \ -policyname app01_policy \ -ruleindex 2 \ -newruleindex 5 # Delete a specific rule cluster01::> vserver export-policy rule delete \ -vserver svm-prod \ -policyname app01_policy \ -ruleindex 2
🧠
When to Add vs Create New Rule
Add to existing rule when the new server needs the exact same access type (same RO/RW/superuser settings) as an existing rule's clients. Create a new rule when the server requires different access levels — adding it to an existing rule would inadvertently grant it the wrong permissions.

📋 Policy-Level Operations

ONTAP CLI — Policy Operations
# Copy an existing policy to a new policy (great for slight variations) cluster01::> vserver export-policy copy \ -vserver svm-prod \ -policyname app01_policy \ -destination-vserver svm-prod \ -destination-policyname app01_policy_v2 # Rename a policy cluster01::> vserver export-policy rename \ -vserver svm-prod \ -policyname app01_policy_v2 \ -newpolicyname app01_policy_dr # Delete a policy (ensure it's unassigned first) cluster01::> vserver export-policy delete \ -vserver svm-prod \ -policyname app01_policy_dr # Verify all policies cluster01::> vserver export-policy show -vserver svm-prod
🚀
Power Feature: Policy Copy
The copy command is a massive time-saver. If you have a policy with 80 rules and a new requirement needs 75 of those same rules, copy the original policy and remove the 5 rules you don't need — rather than building 75 rules from scratch. This is standard practice in large enterprise deployments.

Bulk Access Strategies

Real-world deployments often involve provisioning storage to dozens or hundreds of servers. Adding individual IPs manually is error-prone and time-consuming. ONTAP offers two elegant approaches for bulk client management.

100+ Servers] --> B{Same Subnet?} B -->|Yes, dedicated subnet| C[Subnet / Gateway Method] B -->|No, mixed subnets| D[Netgroup File Method] C --> E[Add gateway IP to clientmatch
e.g. 10.2.0.0/24] C --> F[All servers in subnet get access] D --> G[Create netgroup file
List all server IPs] D --> H[Upload to web server] D --> I[Download to ONTAP as netgroup] D --> J[Reference netgroup name in rule] style C fill:#1a3a2a,stroke:#3fb950,color:#e6edf3 style D fill:#1a2a3a,stroke:#58a6ff,color:#e6edf3 style F fill:#0d1117,stroke:#3fb950,color:#8b949e style J fill:#0d1117,stroke:#58a6ff,color:#8b949e
Figure 5 — Decision flow for bulk NFS access strategies
🌐 Strategy 1: Subnet / CIDR Match

If all target servers belong to a single dedicated subnet, add the subnet's gateway address (or CIDR notation) as the clientmatch. ONTAP will grant access to every IP within that subnet range.

ONTAP CLI — Subnet Clientmatch
# Grant access to all servers in the 10.2.1.0/24 subnet cluster01::> vserver export-policy rule create \ -vserver svm-prod \ -policyname sap_policy \ -clientmatch 10.2.1.0/24 \ -rorule sys \ -rwrule sys \ -superuser sys \ -protocol nfs3 \ -ruleindex 1
🚨
Subnet Method Prerequisites
Only use subnet matching when the subnet is exclusively dedicated to the target application servers. If other unrelated servers (e.g., SharePoint, DNS, etc.) share the same subnet, they will also receive unintended storage access. Confirm with your network team that the subnet is properly segmented before using this approach.
📦 Strategy 2: Netgroup Files

When servers span multiple subnets, netgroups allow you to maintain a central file listing all client IPs and upload it to ONTAP. This is conceptually powerful but complex in practice.

sequenceDiagram participant Admin participant WebServer participant ONTAP Admin->>Admin: 1. Create netgroup file (hosts format) Admin->>WebServer: 2. Upload file to HTTP/FTP server Admin->>ONTAP: 3. Load netgroup from web server Note over ONTAP: vserver export-policy netgroup load ONTAP-->>Admin: Netgroup downloaded & parsed Admin->>ONTAP: 4. Reference netgroup in export rule clientmatch ONTAP-->>Admin: Rule applied — all listed IPs granted access
Figure 6 — Netgroup file workflow
Netgroup File Format
# Example netgroup file: sap_servers sap_servers (app-server-01.corp.local,10.2.1.21,) \ (app-server-02.corp.local,10.2.1.22,) \ (db-server-01.corp.local,10.3.1.31,) \ (db-server-02.corp.local,10.3.1.32,) # Nested netgroup example sap_app_grp (app-server-01,,) (app-server-02,,) sap_db_grp (db-server-01,,) (db-server-02,,) sap_all sap_app_grp sap_db_grp
ONTAP CLI — Netgroup Commands
# View existing netgroups cluster01::> vserver export-policy netgroup show -vserver svm-prod # Load netgroup from web server cluster01::> vserver export-policy netgroup load \ -vserver svm-prod \ -source http://web-server.corp.local/netgroups/sap_servers # Use netgroup in export rule clientmatch cluster01::> vserver export-policy rule create \ -vserver svm-prod \ -policyname sap_policy \ -clientmatch @sap_servers \ -rorule sys \ -rwrule sys \ -superuser sys \ -ruleindex 1 # Delete a netgroup cluster01::> vserver export-policy netgroup delete \ -vserver svm-prod \ -name sap_servers
⚠️
Netgroup Limitations
Netgroup files must be under 5 MB and individual lines cannot exceed 1024 characters. A single syntax error (missing comma, bracket) will cause the entire netgroup to fail silently. Only one netgroup file can be loaded per SVM. In practice, netgroups are rarely used in enterprise environments — subnet-based rules or individual IPs are preferred for their simplicity and reliability.

SECTION 07

Access Verification & Diagnostics

When a Linux team reports that they cannot mount a volume or that access is being denied, ONTAP provides powerful built-in commands to diagnose the issue from the storage side without needing to log into the client server.

✅ Check Access Command

This is the most powerful diagnostic tool for NFS export policy issues. It simulates whether a specific client IP would receive access to a volume and reveals exactly which rule is granting or denying access.

ONTAP CLI — Check Access
# Check if a client has access to a specific volume cluster01::> vserver export-policy check-access \ -vserver svm-prod \ -volume vol_app01 \ -client-ip 10.1.1.21 \ -authentication-method sys \ -protocol nfs3 \ -access-type write Policy Policy Rule Access Vserver Volume Policy Name Owner Index Type ------- ----------- ----------- -------- ----- ------ ------ svm-prod vol_app01 app01_policy volume 1 write allow # Check access for a server that has been removed from policy cluster01::> vserver export-policy check-access \ -vserver svm-prod \ -volume vol_app01 \ -client-ip 10.1.1.99 \ -authentication-method sys \ -protocol nfs3 \ -access-type write Access denied — No matching rule found in policy app01_policy

👥 View Connected NFS Clients

Available from ONTAP 9.7+, this command shows all currently connected NFS clients — which server is accessing which volume, which protocol version, and idle time.

ONTAP CLI — Connected Clients (ONTAP 9.7+)
cluster01::> vserver nfs connected-clients show -vserver svm-prod Node VServer Client IP Volume Protocol Idle Time ---------- ---------- ----------- ----------- -------- --------- node-01 svm-prod 10.1.1.21 vol_app01 NFS3 0:00:05 node-01 svm-prod 10.1.1.21 vol_db01 NFS3 0:01:32 node-01 svm-prod 10.1.1.22 vol_app01 NFS3 0:00:10

⚙️ NFS Service Options

ONTAP CLI — NFS Service Management
# View current NFS configuration cluster01::> vserver nfs show -vserver svm-prod # Enable additional NFS versions cluster01::> vserver nfs modify -vserver svm-prod \ -v4.0 enabled -v4.1 enabled # Stop and start NFS service cluster01::> vserver nfs stop -vserver svm-prod cluster01::> vserver nfs start -vserver svm-prod # Check NFS service status cluster01::> vserver nfs status -vserver svm-prod # Validate export policy config (check for errors) cluster01::> vserver export-policy config-checker start -vserver svm-prod -policyname app01_policy cluster01::> vserver export-policy config-checker show -vserver svm-prod -policyname app01_policy Job rate: success Rules checked: 2 Rules with issues: 0

🔧 Creating Local User Accounts

ONTAP CLI — Local User Creation

🖥️ SSH Application
  • Provides command-line access via SSH clients
  • Standard access method for ONTAP CLI
  • Required for all automation/scripting
  • Works with password or public-key auth
🌐 HTTP + ontapi Application
  • Provides System Manager GUI access
  • Both http AND ontapi must be added
  • Without these, GUI login will fail
  • Use for non-technical administrators

🎭 Creating Custom Roles

Beyond the built-in admin and readonly roles, you can define custom roles that grant precise, granular access to only specific ONTAP commands — perfect for operators who need limited privileges.

ONTAP CLI — Custom Role Creation
# Create a custom role with specific command permissions cluster01::> security login role create \ -role storage_viewer \ -vserver cluster01 \ -cmddirname "volume" \ -access readonly # Add more commands to the role (aggregate read-only) cluster01::> security login role create \ -role storage_viewer \ -vserver cluster01 \ -cmddirname "storage aggregate" \ -access readonly # Add NFS/CIFS read access cluster01::> security login role create \ -role storage_viewer \ -vserver cluster01 \ -cmddirname "vserver cifs" \ -access all # Create a user with this custom role cluster01::> security login create \ -username ops_team_lead \ -application ssh \ -authentication-method password \ -role storage_viewer \ -vserver cluster01
🎯
Custom Role Use Cases
Custom roles are powerful for compliance-driven environments. A support engineer might need to run volume and aggregate show commands but should never be able to delete or create objects. A DBA team may need vserver export-policy access but nothing else. Define the minimum set of commands each persona needs and build a dedicated role — this adheres to the principle of least privilege.

SECTION 09

Active Directory Integration

In enterprise environments with 10, 20, or 50+ ONTAP clusters, managing local accounts becomes an administrative nightmare. Active Directory integration centralizes user management — add a person to an AD group once, and they gain access to all clusters that reference that group. Remove them from AD when they leave, and access is revoked across the board instantly.

Figure 8 — AD integration architecture: domain tunnel, group-based access, and multi-cluster scalability
🔗 Domain Tunnel — The Bridge to Active Directory

A domain tunnel creates a secure connection between the ONTAP cluster and the Active Directory domain controller through an SVM where CIFS is configured. This is the prerequisite for any AD-based authentication.

ONTAP CLI — Domain Tunnel Setup
# Prerequisites: Ensure CIFS is configured on at least one SVM cluster01::> vserver show -type data # Check existing domain tunnels cluster01::> security login domain-tunnel show # Create the domain tunnel (use SVM where CIFS is configured) cluster01::> security login domain-tunnel create -vserver svm-cifs # Verify tunnel is created cluster01::> security login domain-tunnel show Vserver ----------- svm-cifs

🔐 Adding AD Users & Groups

ONTAP CLI — AD User & Group Configuration
# Add individual AD user — SSH + GUI access cluster01::> security login create \ -username corp\alice \ -application ssh \ -authentication-method domain \ -role admin \ -vserver cluster01 cluster01::> security login create -username corp\alice -application http -authentication-method domain -role admin cluster01::> security login create -username corp\alice -application ontapi -authentication-method domain -role admin # Add AD group (recommended — scales across all members) cluster01::> security login create \ -username corp\SAN-Admins \ -application ssh \ -authentication-method domain \ -role admin \ -vserver cluster01 # Verify all users including AD accounts cluster01::> security login show Vserver UserName App Auth-Method Role ------- --------------- ------- ----------- -------- cluster admin ssh password admin cluster corp\alice ssh domain admin cluster corp\SAN-Admins ssh domain admin ... # Verify which account you're logged in with cluster01::> security login whoami
🌟
Production Best Practice: Group-Based AD Access
Never add individual AD users to ONTAP clusters if you can avoid it. Instead, create an AD group (e.g., SAN-Admins) and add that group to all clusters once. Onboarding a new engineer means simply adding them to the AD group — no ONTAP changes across 50 clusters. Offboarding means removing from the AD group — instant revocation everywhere. This is the standard approach at all major enterprises.
⚠️
Domain Accounts — No Password Prompt in ONTAP
When creating domain-authenticated accounts (-authentication-method domain), ONTAP will not ask you to set a password. The password is managed entirely by Active Directory. This is by design — credential control stays with the AD team.

SECTION 10

Scenario-Based Interview Q&A

❓ Q1: A Linux admin says they cannot mount a volume but you see the export policy rule is configured. How do you diagnose this?
Use vserver export-policy check-access specifying the client IP, volume name, protocol (nfs3), and authentication method (sys). This command simulates whether the client would be granted access and shows the exact policy rule that is matching (or not). Also confirm: (1) superuser is set to sys in the rule, (2) the junction path exists on the volume, (3) the LIF IP or DNS name is reachable from the client.
❓ Q2: You need to provision NFS access to 300 Linux servers. They all belong to the same subnet (10.5.1.0/24). What is the most efficient approach?
Use subnet-based clientmatch in the export policy rule — specify -clientmatch 10.5.1.0/24. This grants access to all IPs within that /24 subnet with a single rule. This is the preferred method when all servers belong to a dedicated, application-specific subnet. Verify with the network team that no unrelated servers share this subnet.
❓ Q3: Your organization has 15 ONTAP clusters. A new storage engineer has joined. What is the most scalable way to grant them access?
If AD integration is configured, add the engineer to the existing AD group (e.g., SAN-Admins) that is already linked to all 15 clusters. They will immediately gain access to all clusters via their domain credentials — zero ONTAP configuration changes needed. If AD is not configured, you'd need to create a local account on all 15 clusters individually, which is why AD groups are strongly preferred in multi-cluster environments.
❓ Q4: What is the difference between read-only and read-write access in export policy rules? Why is superuser a separate field?
Read-only (RO) allows reading files/directories; read-write (RW) allows creating, modifying, and deleting. Superuser is a separate concept representing root-level OS access — without it set to sys, the client's root user cannot mount the volume at all, regardless of RO/RW settings. The superuser field controls whether the Linux root account's NFS requests are treated as root on the storage side, which is essential for the mount operation itself.
❓ Q5: You have an export policy with 120 rules for a production volume. You need to onboard a new application that is identical but requires access for only 115 of those servers. How do you handle this efficiently?
Use vserver export-policy copy to duplicate the existing 120-rule policy to a new policy name. Then delete the 5 rules that the new application does not need. Apply the new policy to the new volume. This avoids manually recreating 115 rules from scratch — a major time saving in large-scale environments.
❓ Q6: What is a domain tunnel in NetApp ONTAP and why is it required for AD integration?
A domain tunnel is a configuration entry that designates a specific SVM (which has CIFS configured and is joined to the AD domain) as the conduit for cluster-level AD authentication. The ONTAP cluster itself does not join AD — instead, it leverages the SVM's existing AD membership through this tunnel to validate domain credentials. Without it, ONTAP cannot reach the domain controller to authenticate AD users or groups, even if the CIFS SVM exists.
