SECTION 01

What Is a NetApp Cluster?

A cluster is a logical grouping of storage controllers (nodes) that work together as a unified system. Modern ONTAP Cluster Mode (cDOT) scales far beyond the legacy 7-Mode limit of one or two nodes.

1–2
7-Mode Node Limit
24
Max Nodes (Cluster Mode)
12
Maximum HA Pairs
ℹ️
Cluster vs 7-Mode
NetApp 7-Mode is a legacy single-controller architecture. Cluster Mode (ONTAP 8.x+) introduces a distributed, scale-out model. All modern deployments use Cluster Mode.

📌 Node Count Rules

⚡ The Even-Number Constraint

Every node must belong to an HA pair — odd node counts are not supported. Valid configurations:

2 nodes 4 nodes 6 nodes 8 nodes ... up to 24 ✗ 3 nodes ✗ 5 nodes ✗ 7 nodes
⚠️
HA Failover Scope
Failover happens only within the HA pair. If Node A fails, Node B takes over — Node C or D from another pair does not participate.

🧩 Mixing Node Models

✅ Allowed
  • FAS systems of different series
  • AFF systems of different series
  • FAS and AFF together in the same cluster
❌ Not Allowed
  • ASA cannot be mixed with FAS or AFF
  • Nodes within the same HA pair must be identical models
💡
Critical — HA Pair Model Matching
Both nodes within an HA pair must be identical hardware models. All nodes in a cluster must run the same ONTAP version.

SECTION 03

Cluster Interconnect Network

The Cluster Interconnect is the private, high-speed backbone for inter-node communication — handling both configuration synchronization and data traffic.

🔌 Port Speed Requirements

Each node requires a minimum of two dedicated cluster interconnect ports:

10 GbE (×4 ports) 25 GbE (×2 ports) 40 GbE (×2 ports) 100 GbE (×2 ports)

Cluster interconnect ports are pre-configured by NetApp and auto-assign IPs in the 169.254.x.x/16 link-local range — no manual assignment needed.

Figure 2 — Switched vs Switchless cluster selection decision tree

Option A: Switchless Cluster (Two-Node Only)

Cluster interconnect ports are connected directly between Node 1 and Node 2 — no switches required. Link-local 169.254.x.x addresses are automatically assigned.

⚠️
HA Interconnect ≠ Cluster Interconnect
The internal HA interconnect (typically 100 MB, used for heartbeat) is not suitable for cluster traffic. Cluster interconnect must use dedicated 10/25/40 GbE ports.

Option B: Switched Cluster (2–24 Nodes)

Mandatory for three or more nodes. Uses certified, dedicated network switches to form the cluster interconnect fabric.

Figure 3 — Switched cluster: each node connects one port to each switch (dual-homed)

Migrating Switchless → Switched (Non-Disruptive)

  • 1Procure two certified cluster switches and rack them.
  • 2While Port 1 cross-connect remains active, disconnect Port 2 from both nodes.
  • 3Connect Port 2 on each node to Switch 1 and Switch 2 respectively.
  • 4Verify cluster communication via the new switch path.
  • 5Disconnect Port 1 cross-connect and connect to the switches.
  • 6Configure ISL (Inter-Switch Links) between the two cluster switches.
🚨
Critical: One Cable at a Time
Never disconnect both cluster interconnect links simultaneously — this breaks cluster communication entirely. Always maintain at least one active path during migration.

🔩 Shaft & Platters

The shaft is the central axis; all platters spin together around it. Data is magnetically encoded onto platter surfaces in concentric tracks.

📍 Read/Write Head & Sectors

The head floats nanometers above the platter, writing and reading data as the platter spins. Sectors are the smallest addressable storage units on each platter.

SECTION 08

Data Writing Techniques: Concatenation vs Striping

RAID types are built on two foundational data distribution strategies:

block-beta columns 7 block:concat_title:3 space A["🔄 CONCATENATION"] space end space block:stripe_title:3 space B["⚡ STRIPING"] space end block:disk1_c:1 C1["Disk 1\n███████\nFull of A"] end block:disk2_c:1 C2["Disk 2\n███████\nFull of B"] end block:disk3_c:1 C3["Disk 3\n███\nPartial C"] end space block:disk1_s:1 S1["Disk 1\n█A █D █G"] end block:disk2_s:1 S2["Disk 2\n█B █E █H"] end block:disk3_s:1 S3["Disk 3\n█C █F █I"] end
Figure 6 — Concatenation fills disk-by-disk; Striping distributes across all disks simultaneously
🔄 Concatenation

Data fills Disk 1 completely, then overflows to Disk 2. Only one disk is active at a time — poor performance, uneven wear.

⚡ Striping

Data is distributed in stripes across all disks simultaneously — all disks work in parallel, maximizing throughput.

⚡
NetApp Uses Striping Exclusively
ONTAP automatically applies striping across all aggregates. Concatenation is not an option — NetApp determined striping is always the superior choice for storage workloads.
🛡️
SECTION 09

RAID Types in NetApp ONTAP

RAID (Redundant Array of Independent Disks) protects data when physical drives fail. ONTAP supports four RAID types, each offering different fault tolerance and capacity efficiency.

📌
RAID Is Mandatory in ONTAP
Every aggregate must have a RAID policy. You cannot create a storage aggregate without specifying a RAID type at creation time.
Figure 7 — RAID types plotted by fault tolerance vs capacity efficiency

RAID 0 — Striping Without Protection

📊 RAID 0 Specifications
Disk 1
A · D · G
Disk 2
B · E · H
Disk 3
C · F · I

All disks are data disks — no parity

ProtectionNone
Fault Tolerance0 disks
Min Disks (cMode)2
Usable Capacity100% (N)

RAID 0 provides zero data protection — a single disk failure causes complete data loss. Its only use in ONTAP is for Flash Pool cache aggregates, where data already exists on a protected spinning disk layer.

RAID 4 — Single Parity Protection

📊 RAID 4 Specifications
Stripe 1: A B A+B
Stripe 2: C D C+D
Data Data Parity
ProtectionSingle Parity
Fault Tolerance1 disk
Min Disks (cMode)3
Usable CapacityN−1

One dedicated parity disk stores an XOR checksum of each data stripe, enabling recovery of any single failed disk. Through XOR mechanics, one parity disk of equal size can represent all data disk checksums — it stores logical checksums, not copies of data.

🔧
Parity Reconstruction Logic
If Disk 1 (data=A) fails: Parity − B = A (reconstructed). If the Parity disk fails: A + B = new Parity. ONTAP handles all reconstruction automatically.

RAID-DP — Dual Parity (The Default)

📊 RAID-DP Specifications (Dual/Diagonal Parity)
Stripe 1: A B C P1 DP1
Stripe 2: D E F P2 DP2
D D D P DP

P = Row Parity · DP = Diagonal Parity

ProtectionDual Parity
Fault Tolerance2 disks
Min Disks (cMode)5
Min Disks (7-Mode)3
Usable CapacityN−2

RAID-DP is NetApp's default RAID type, maintaining two dedicated parity disks per RAID group: a Row Parity (P) disk using XOR across each stripe, and a Diagonal Parity (DP) disk computed diagonally across multiple stripes. This two-dimensional scheme survives any two simultaneous disk failures.

🔷 NetApp RAID-DP
  • Two dedicated parity disks (fixed positions)
  • Higher write performance due to parity isolation
  • Parity disks see heavier write load
🔷 RAID 6 (Other Vendors)
  • Parity distributed across all disks per stripe
  • Even write load distribution across all disks
  • Better performance under balanced load
📝
Interview Insight: RAID-DP vs RAID 6
Key distinction: RAID-DP parity sits on fixed dedicated disks; RAID 6 parity rotates per stripe. Same protection goal, different mechanical approach. NetApp does not implement RAID 6.

RAID-TEC — Triple Parity for Massive Drives

📊 RAID-TEC Specifications (Triple Parity)
Stripe 1: A B C D P1 DP1 TP1
D D D D P DP TP

TP = T-Parity (reverse diagonal)

ProtectionTriple Parity
Fault Tolerance3 disks
Min Disks (cMode)7
Usable CapacityN−3

RAID-TEC adds a third parity dimension — T-Parity — computed in a reverse diagonal direction, enabling survival of up to three simultaneous drive failures. Designed for very high-capacity drives (8 TB+) where reconstruction windows are long enough to raise the statistical risk of a second or third failure.

💡
When to Choose RAID-TEC
Use RAID-TEC when individual disk capacity is 8 TB or larger, reconstruction time is measured in hours, or three-disk fault tolerance is required for compliance. RAID-TEC also offers faster reconstruction than RAID-DP due to newer parity algorithms.

SECTION 10

Disk Reconstruction — How ONTAP Recovers Data

When a disk fails, ONTAP automatically initiates reconstruction. No admin commands needed — the system detects the failure and begins recovering lost data onto a spare or replacement disk.

Figure 8 — Automated disk reconstruction workflow in NetApp ONTAP
🔧 Reconstruction Mechanics
  • Data disk failure: ONTAP uses parity to reconstruct lost data block-by-block. E.g., if Disk A failed: Parity − Disk B = Reconstructed A.
  • Parity disk failure: Parity is recalculated by summing all data disks — no data is at risk.
  • Hot spare: Most environments maintain a spare disk. Reconstruction starts immediately upon failure — no manual intervention.
⚠️
Reconstruction Window Risk
During reconstruction, RAID-DP temporarily operates with only 1-disk fault tolerance instead of 2. For large-capacity drives with long rebuild times, this is why RAID-TEC is preferred — it maintains 2-disk tolerance even during a rebuild.

SECTION 11

Interview Q&A

QWhy must clusters always have an even number of nodes?
Every node must be part of an HA pair to have a failover partner. Without a partner, there is no node to take over services during a failure. ONTAP's architecture mandates HA-pair membership for all nodes, enforcing even node counts.
QA customer wants a 6-node cluster using two FAS9500 HA pairs and one AFF A800 HA pair. Is this supported?
Yes — mixed FAS and AFF hardware is supported when: (1) each HA pair contains identical model controllers, (2) ASA systems are not included, and (3) all nodes run the same ONTAP version.
QWhat is the difference between the HA interconnect and the cluster interconnect?
The HA interconnect is a low-bandwidth internal link (~100 MB) for heartbeat and failover signaling within an HA pair. The cluster interconnect is a high-bandwidth network (10/25/40/100 GbE) for data traffic and configuration replication between all cluster nodes. They serve fundamentally different purposes and cannot substitute for each other.
QWhen would you recommend RAID-TEC over RAID-DP?
RAID-TEC is recommended for aggregates using 8 TB+ drives, where reconstruction after failure takes many hours. The longer rebuild window increases the statistical risk of a second failure. RAID-TEC's triple parity provides an additional protection layer during reconstruction, and its optimized parity algorithms also deliver faster rebuild times.
QA junior engineer says RAID 6 is better than RAID-DP. How do you respond?
RAID 6 and RAID-DP offer identical fault tolerance (two simultaneous disk failures). The difference is parity placement: RAID 6 distributes parity per stripe across all disks (better load balancing); RAID-DP uses dedicated parity disks (simpler design, optimized for ONTAP's WAFL architecture). Neither is categorically better — it depends on the implementation. NetApp uses RAID-DP, not RAID 6.
QWalk me through migrating a two-node switchless cluster to switched without downtime.
Procure two certified switches. While Port 1 cross-connect remains active, disconnect Port 2 on both nodes and connect to the switches. Verify cluster communication, then disconnect Port 1 and connect to the switches. Configure ISL between switches. Finally, join the new HA pair. Never remove both direct cables simultaneously — this breaks cluster communication entirely.
QA DBA needs 4,200 IOPS from Oracle. How do you size the disk pool?
With SAS 15K at ~175 IOPS each: 4,200 ÷ 175 = 24 drives for raw IOPS. Account for RAID-DP (2 parity disks) and at least 1 spare. Total recommendation: 27 SAS 15K drives (24 data + 2 parity + 1 spare). If the workload is latency-sensitive OLTP, also recommend evaluating SSDs despite higher cost.
QWhat is NLSAS and how does it relate to SATA?
NLSAS (Near-Line SAS) and BSAS (Bridged SAS) are NetApp's internal naming conventions for SATA drives. The underlying technology is SATA — NetApp uses its own terminology for internal classification in ONTAP. They are the same physical drive type.
