SECTION 02

CIFS Shares — Fundamentals

A CIFS share (also referred to as an SMB share) is the mechanism by which ONTAP presents a volume or qtree to Windows clients. Think of a share as the "front door" — it defines what storage space is accessible over the network and under what name. When a business user requests "a 500 GB shared drive for the Finance team," what they are asking for is a CIFS share backed by an ONTAP volume.

💡
Key Concept
Before creating a CIFS share, the underlying volume must have its security style set to NTFS and a junction path configured. Without a junction path, the volume is not accessible through the NAS namespace and cannot be shared.

Creating a Volume and CIFS Share

The typical workflow is: create an NTFS-style volume → assign a junction path → create the CIFS share pointing to that path.

ONTAP CLI — Volume + Share Creation

⚠️
Default Shares
ONTAP automatically creates several system shares (admin$, ipc$, c$) when a CIFS server is configured. These are administrative shares — do not delete them. Only the share you explicitly create (like Finance_Data above) is yours to manage.

Share-Level Permissions

CIFS provides two layers of access control. Share-level permissions are the outer gate — a user must pass share-level permissions before ONTAP even evaluates NTFS (file-level) permissions. ONTAP supports four permission types:

🚫 No Access

Explicitly denies access regardless of group membership. Even domain admins with this setting cannot enter the share.

👁️ Read

View and copy files only. Cannot create, modify, or delete data. Ideal for report consumers or auditors.

✏️ Change

Full data access — read, write, modify, and delete. Cannot manage permissions on the share itself.

👑 Full Control

All Change privileges plus permission management. The user can grant/revoke other users' access. Default for "Everyone" on share creation.

🔴
Critical — Remove Everyone/Full Control Immediately
When a CIFS share is first created, Everyone gets Full Control by default. This means anyone who knows the share name and virtual CIFS server name can access it without authentication. Always restrict this immediately after share creation by adding specific users or AD groups, then deleting the "Everyone" ACL entry.
ONTAP CLI — Managing Share Access Control
# Grant access — Full Control to a specific AD user ::> vserver cifs share access-control create \ -vserver svm-finance \ -share Finance_Data \ -user-or-group corp\sarah.johnson \ -permission Full_Control # Grant access — Read only to an AD group ::> vserver cifs share access-control create \ -vserver svm-finance \ -share Finance_Data \ -user-or-group corp\Finance_Readers \ -permission Read # Grant Change access to another user ::> vserver cifs share access-control create \ -vserver svm-finance \ -share Finance_Data \ -user-or-group corp\mark.chen \ -permission Change # Remove "Everyone" — do this LAST after adding named users ::> vserver cifs share access-control delete \ -vserver svm-finance \ -share Finance_Data \ -user-or-group Everyone # Modify existing permission (no need to delete and recreate) ::> vserver cifs share access-control modify \ -vserver svm-finance \ -share Finance_Data \ -user-or-group corp\mark.chen \ -permission Read # View current share ACLs ::> vserver cifs share show -vserver svm-finance

Connecting to a CIFS Share — Windows Client

Windows users can access CIFS shares in two ways: direct path navigation (\\CIFS_SERVER\share_name) or mapping as a persistent drive letter via "Map Network Drive." Using the virtual CIFS server DNS name (rather than an IP address) is strongly recommended — DNS is mapped to all data LIF IPs, providing automatic failover if one LIF becomes unreachable.

SECTION 03

NTFS / User-Level Permissions

Beyond share-level permissions, ONTAP volumes with NTFS security style support file and folder-level ACLs (Access Control Lists), also known as NTFS permissions. These are the same permission mechanism used by Windows NTFS volumes and are managed through the Windows Security tab in file or folder properties.

📋
Who Manages What
In most organizations: Storage team manages share-level permissions (the outer gate). Windows/AD team manages NTFS permissions (the inner gate). Storage admins typically add the Windows team's AD group to the share with Full Control, then the Windows team handles individual file/folder ACLs via Computer Management or Windows Explorer.

Managing NTFS Permissions via Computer Management

Administrators can connect to the ONTAP SVM from the Windows Computer Management console and manage shares and NTFS permissions directly, just as they would for a local Windows server. To connect:

  1. Open Computer Management on a Windows server
  2. Right-click → Connect to another computer
  3. Enter the virtual CIFS server DNS name (e.g., svm-finance.corp.example.com)
  4. Expand System Tools → Shared Folders → Shares
  5. Right-click any share → Properties → Security tab for NTFS ACLs
⚠️
Two-Layer Access Model
A user must satisfy both share-level AND NTFS-level permissions to access data. If a user has share-level access but no NTFS permission on a folder, they'll be denied. Conversely, NTFS permissions are mandatory — even if share-level access is missing, NTFS permissions still apply at the file system level. Always configure both layers.

Access-Based Enumeration (ABE)

Access-Based Enumeration is one of the most powerful security features available in NetApp CIFS. Without ABE, when a user browses a share, they can see all folders regardless of whether they have permission to access them — this gives away information about what data exists. With ABE enabled, users can only see folders and files they are permitted to access. Hidden folders simply don't appear in their view.

ABE hides folders from users who lack NTFS access — each user sees only their permitted folders
✅
Best Practice — Enable ABE on Shared Departmental Shares
ABE is especially important for shares where multiple teams store data in separate folders (HR, Finance, Legal, etc.). Without ABE, a curious user can see folder names — a security and compliance risk. ABE is added as a share property using vserver cifs share properties add.

SECTION 04

Share Properties Deep Dive

Share properties are optional features you can attach to a CIFS share. ONTAP applies a set of defaults automatically (oplocks, browsable, change-notify, show-previous-versions), but you can fully customize what is applied. Properties can be added or removed at any time, even after share creation.

PropertyDefaultDescriptionUse When
oplocksYesOpportunistic locks — clients cache data locally for performanceMost general-purpose shares
browsableYesShare is visible when browsing the CIFS server (\\svm-name)Standard shares; remove to hide confidential shares
changenotifyYesNotifies clients of file system changes in real timeMost shares; remove only if creating performance issues
show-previous-versionsYesEnables Windows "Previous Versions" restore tab (requires snapshots)When snapshots are configured on the volume
access-based-enumerationNoHide folders/files the user cannot accessMulti-tenant or multi-department shares
continuously-availableNoPersistent session IDs for transparent reconnectionHyper-V over SMB3 or SQL Server over SMB3 only
namespace-cachingNoCache directory structure client-side for faster enumerationLarge directory trees
ONTAP CLI — Share Properties Management
# Create share with custom options upfront ::> vserver cifs share create \ -vserver svm-finance \ -share-name Exec_Reports \ -path /exec_data \ -comment "Confidential — C-Suite use only" \ -vscan-fileop-profile standard \ -max-connections-per-share 10 \ -share-properties oplocks,changenotify,access-based-enumeration # Add a property to an existing share ::> vserver cifs share properties add \ -vserver svm-finance \ -share-name Finance_Data \ -share-properties access-based-enumeration # Remove browsable to hide a share ::> vserver cifs share properties remove \ -vserver svm-finance \ -share-name Exec_Reports \ -share-properties browsable # Modify share settings post-creation ::> vserver cifs share modify \ -vserver svm-finance \ -share-name Finance_Data \ -vscan-fileop-profile writes-only \ -max-connections-per-share 25 # Delete a share (volume is preserved) ::> vserver cifs share delete \ -vserver svm-finance \ -share-name Finance_Data

Antivirus Scanning (VScan Profile)

ONTAP supports integration with external antivirus scanning engines via the VScan framework. Rather than copying data to a scanning server, the scan happens inline — data is inspected byte-by-byte during the write or read operation, over the network. Each CIFS share has its own scan profile that controls this behavior.

VScan ProfileBehaviorPerformance Impact
standardScans reads and writes; may skip files on timeoutModerate — recommended default
strictScans everything — no skips, no timeout bypassHigh — use for highly sensitive data only
writes-onlyScans new incoming writes; reads bypass scanningLower — good compromise for performance-sensitive shares
no-scanDisables antivirus scanning entirely for this shareNone — use only for trusted data sources
💜
Interview Tip — Why is SMB slower than other protocols?
SMB/CIFS is typically slower than NFS or iSCSI for high-throughput workloads primarily because of inline antivirus scanning. Every SMB write operation passes through the VScan engine by default. This overhead is why SMB is not recommended for database or high-IOPS workloads. For those, use NFS or block (iSCSI/FC).

Continuous Availability — When and Why

SMB is a stateful protocol — it maintains connection state (session IDs, connection IDs). If a network interruption occurs or a client is idle for a period, the session is dropped and a new session ID is assigned on reconnect. For most file sharing workloads, this is acceptable. For Hyper-V virtual machines and SQL Server databases running directly over SMB3, this session disruption causes VM/database errors.

Enabling the Continuously Available (CA) share property allocates a persistent session ID for that share. If a brief network disruption occurs, the client reconnects using the same session ID — the application sees no interruption. ONTAP explicitly warns against using CA for general-purpose shares due to resource overhead.

sequenceDiagram participant VM as Hyper-V VM participant S as SMB3 Share (CA enabled) participant O as ONTAP SVM VM->>S: Connect (Session ID: 7A2F) S->>O: Register persistent session Note over VM,O: Brief network interruption... VM->>S: Reconnect attempt S->>O: Resume session ID: 7A2F O-->>S: Session restored, same ID S-->>VM: ✅ Transparent reconnect — no disruption
Continuously Available shares maintain persistent session IDs — critical for Hyper-V and SQL over SMB3
🔴
Warning — Do Not Use CA on General Shares
ONTAP itself issues a warning when you enable Continuously Available on a non-Hyper-V/SQL share. CA shares consume additional controller resources to maintain persistent sessions. For standard file shares, this is wasted overhead. Use CA only for SMB3 Hyper-V or SQL Server integrations.

SECTION 05

CIFS Server Management

Beyond share management, ONTAP provides a rich set of commands to administer the virtual CIFS server itself — from creating NetBIOS aliases and monitoring active connections to controlling SMB protocol versions and managing domain controller preferences.

NetBIOS Alias

A NetBIOS alias lets you add an alternate name to your virtual CIFS server without modifying the original configuration. This is useful when teams know the server by a legacy name, or when you want to migrate from an old server name gradually without disrupting existing mapped drives.

ONTAP CLI — NetBIOS Alias
# Add a NetBIOS alias to the virtual CIFS server ::> vserver cifs add-netbios-aliases \ -vserver svm-finance \ -netbios-aliases FIN-SERVER,FINANCE-NAS # Verify — look for "NetBIOS Aliases" in the output ::> vserver cifs show -vserver svm-finance -instance # Remove an alias when no longer needed ::> vserver cifs remove-netbios-aliases \ -vserver svm-finance \ -netbios-aliases FIN-SERVER

Connections, Sessions & File Locks

ONTAP provides granular visibility into every active SMB connection and session. This is invaluable during maintenance windows (to verify no users are connected before a scheduled operation) and for troubleshooting access issues.

ONTAP CLI — Connections, Sessions & File Locks
# View all active CIFS connections ::> vserver cifs connection show \ -vserver svm-finance Node Vserver Conn-ID Session-ID Remote-IP Port LIF -------- --------- -------- ----------- -------------- ----- ---------- node-01 svm-finance 4732 2819 172.16.10.101 445 data-lif-01 # View all SMB sessions with idle time and open files count ::> vserver cifs session show -vserver svm-finance # View which files are currently open (great for troubleshooting locks) ::> vserver cifs session file show -vserver svm-finance # Force-close a stale file lock (use conn-id and session-id from above) ::> vserver cifs session file close \ -vserver svm-finance \ -node node-01 \ -session-id 2819 \ -connection-id 4732 \ -file-id 32 # Force-close an idle session to free up connection slots ::> vserver cifs session close \ -vserver svm-finance \ -node node-01 \ -session-id 2819 \ -connection-id 4732 # Stop and start the virtual CIFS server ::> vserver cifs stop -vserver svm-finance ::> vserver cifs start -vserver svm-finance
💜
Real-World Scenario — Releasing a File Lock
A team member opened a shared Excel report before leaving for a business trip. Colleagues on a different time zone cannot save their updates because the file is locked under that user's session. Use vserver cifs session file show to identify the file ID and session, then vserver cifs session file close to release the lock. Any unsaved changes by the original user will be lost, so confirm this is acceptable before proceeding.

Domain Controller Management

The CIFS server must communicate with Active Directory domain controllers (DCs) for every user authentication. ONTAP automatically discovers all DCs in the domain, but you can specify preferred DCs to prioritize local domain controllers (same datacenter) over geographically remote ones — reducing authentication latency significantly.

SMB Protocol Versions — Control & Security

ONTAP supports SMB 1, SMB 2, SMB 3, and SMB 3.1. Understanding which versions to enable is a critical security decision. SMB 1 and SMB 2 have known vulnerabilities that were exploited by ransomware attacks (WannaCry, NotPetya). SMB 3 and 3.1 include encryption and pre-authentication integrity — significantly more secure.

VersionDefault (ONTAP 9.3+)Notes
SMB 1DisabledRansomware-vulnerable. Do not enable unless required for legacy systems (e.g., Windows XP era devices).
SMB 2EnabledWidely compatible, but lacks encryption. Use only if SMB 3 is not supported by the client OS.
SMB 3EnabledEnd-to-end encryption, persistent handles. Required for Hyper-V and SQL over SMB.
SMB 3.1EnabledPre-authentication integrity (prevents man-in-the-middle). Recommended standard for modern environments.
ONTAP CLI — SMB Version Control (Advanced Mode)
# Enter advanced privilege mode ::> set advanced # View and modify CIFS options (SMB versions, security settings) ::*> vserver cifs options modify \ -vserver svm-finance \ -smb1-enabled false \ -smb2-enabled true \ -smb3-enabled true \ -smb31-enabled true # Verify current SMB version settings ::*> vserver cifs options show \ -vserver svm-finance # Return to admin mode ::*> set admin
🔴
Security Advisory — Disable SMB 1 Immediately
SMB 1 was the primary attack vector for the WannaCry and NotPetya ransomware outbreaks. ONTAP 9.3 and later disable SMB 1 by default. If you are managing systems running older ONTAP versions, explicitly verify that SMB 1 is disabled. Enabling it even temporarily for legacy device compatibility creates significant organizational risk.

SECTION 06

WAFL — Write Anywhere File Layout

WAFL (Write Anywhere File Layout) is the proprietary file system at the core of ONTAP. Every storage operating system requires a file system to manage how data is physically placed on disks — in Windows it's NTFS, in Linux it's ext4/xfs, and in ONTAP it's WAFL. Understanding WAFL explains why ONTAP behaves differently from traditional storage arrays and why certain design decisions (like the 10% WAFL reserve) exist.

📘
What Does a File System Do?
A file system is responsible for deciding where data is physically written to disk, maintaining metadata (file names, sizes, timestamps), and tracking which blocks are free vs. used. Without a file system, a disk is just a sequence of raw blocks with no meaningful organization.

Traditional File System vs. WAFL

The fundamental difference between WAFL and traditional file systems lies in block reservation. Traditional file systems reserve specific blocks for specific logical partitions at creation time. WAFL does not — it writes wherever the next free block is found.

🔒 Traditional File System
EMC, IBM, Hitachi — legacy approach
🌊 WAFL — Write Anywhere
NetApp ONTAP — modern approach
✅ WAFL Advantages
  • Superior write performance — always writes to the next available free block, eliminating seek delays to reserved areas
  • Efficient space utilization — no pre-allocation waste; thin provisioning works naturally
  • Snapshot efficiency — WAFL's copy-on-write mechanism makes snapshots nearly instantaneous and space-efficient
  • Consistent performance — no fragmentation effects of traditional reserved-block layouts
⚠️ WAFL Tradeoffs
  • Read latency overhead — since blocks for a volume are scattered across the aggregate, reads may scan more locations (mitigated by PAM/FlashCache)
  • 10% WAFL reserve — every aggregate reserves 10% of its capacity for WAFL metadata, Snapshot overhead management, and consistency tracking
  • PAM dependency — to compensate for read scatter, NetApp introduced the Performance Acceleration Module (FlashCache) to cache block location metadata
🔍
Why PAM/FlashCache Exists
WAFL's write-anywhere approach means a volume's data is spread across the entire aggregate. When reading, ONTAP must locate those scattered blocks — adding potential latency. The Performance Acceleration Module (PAM) — a PCIe flash card — caches block location indexes and frequently accessed data in extremely low-latency flash memory. This largely eliminates the read-performance disadvantage of WAFL compared to traditional reserved-block file systems.

SECTION 07

Inodes — File Metadata & Capacity Planning

An inode is a metadata record that ONTAP assigns to every file and folder stored in a volume. The inode stores file attributes: name, size, timestamps, location of data blocks, permissions — everything except the actual file content. Think of inodes as the library catalog entries for your data.

This matters for capacity planning because ONTAP assigns a fixed pool of inodes per volume based on volume size at creation time. If a volume fills up its inode pool — even if raw disk space remains available — no new files can be created until inodes are freed or the pool is expanded.

🔴
Common Production Issue — Inode Exhaustion
A user reports they cannot create new files even though the volume shows free space. Root cause: the inode pool is exhausted. This commonly happens in volumes storing millions of small files (email stores, log archives, audit trails). Always monitor both space utilization AND inode utilization for such workloads.

Inode Capacity Calculation

ONTAP's default inode allocation is 1 inode per 32 KB of volume capacity. You can increase this up to a maximum of 1 inode per 4 KB — the smallest block size ONTAP allocates.

ONTAP CLI — Inode Monitoring & Management
# Method 1 — Check inodes using df (Seven-Mode style, also works in Cluster Mode) ::> df -i /vol/fin_data01 Filesystem iused ifree %iused Mounted on /vol/fin_data01 101 32667 0% /fin_data01 # Method 2 — Cluster Mode command (preferred for ONTAP 9.x) ::> volume show \ -vserver svm-finance \ -volume fin_data01 \ -fields files,files-used vserver volume files files-used ----------- ------------ ------ ---------- svm-finance fin_data01 32768 101 # Increase inode count manually (when running out of inodes but space is available) ::> volume modify \ -vserver svm-finance \ -volume fin_data01 \ -files 200000 # Verify the updated inode pool ::> volume show \ -vserver svm-finance \ -volume fin_data01 \ -fields files,files-used vserver volume files files-used ----------- ------------ ------- ---------- svm-finance fin_data01 200000 101
✅
Best Practice — Inode Capacity Planning
For volumes expected to store large numbers of small files (email archives, document management, log files): estimate the expected file count, divide by volume size to determine KB/inode needed, and pre-configure the inode count at volume creation. Expanding inode count after the fact is supported but requires proactive monitoring. The maximum is 1 inode per 4 KB — beyond that, you must grow the volume.

SECTION 08

Scenario-Based Interview Q&A

Q1 What happens if you delete "Everyone" from share permissions before adding specific users?
The share becomes inaccessible to all users immediately, including the administrator who is making the change. Always add specific users or groups first, verify they appear in the ACL, and then remove "Everyone." This is the correct operational sequence. If Everyone is removed prematurely, only a user with cluster administrator access can restore share permissions via ONTAP CLI.
Q2 A user has Full Control share permission but cannot access a specific folder. What would you check?
Share-level and NTFS-level permissions are independent. Having Full Control at the share level does not bypass NTFS (file-level) permission restrictions. Check the NTFS ACL on that specific folder via Computer Management → Shares → Security tab. The user or their AD group may have no NTFS permission, or may have an explicit Deny entry. Both share permission AND NTFS permission must permit access for the user to successfully open the folder.
Q3 How would you hide a CIFS share without deleting it?
Remove the browsable share property using vserver cifs share properties remove -share-properties browsable. The share will no longer appear when a user browses \\cifs-server-name. However, users who know the exact share name can still access it directly via \\cifs-server-name\share-name if they have the appropriate permissions. This provides obscurity, not security — combine it with strict ACLs for true protection. A common convention is to append a $ suffix to a share name (e.g., exec$) to signal it is hidden.
Q4 What is the difference between Change and Full Control share permissions?
Change (sometimes called Write in older documentation) grants full data access: read, write, create, modify, and delete files and folders within the share. However, the user cannot manage the share's own permission list. Full Control includes everything Change provides, plus the ability to modify share permissions — meaning the user can grant or revoke other users' access to the share. Full Control is the CIFS equivalent of being the share owner. For most regular users, Change is sufficient and safer.
Q5 A team member left for a long trip and forgot to close an Excel file on a CIFS share. Colleagues cannot save updates to that file. How do you resolve this?
Use vserver cifs session file show to identify the file lock — it shows the file ID, session ID, connection ID, volume, and user who holds the lock. Once identified, use vserver cifs session file close with the appropriate IDs to release the lock. The file becomes available for others to save changes. The original user's unsaved work is lost when the lock is released, so always confirm this is acceptable — ideally by contacting the file owner or getting management approval.
Q6 Explain WAFL and why NetApp introduced PAM/FlashCache.
WAFL (Write Anywhere File Layout) is ONTAP's proprietary file system. Unlike traditional file systems that reserve specific disk blocks for specific volumes, WAFL writes data to any available free block in the aggregate. This delivers excellent write performance because the system never has to seek reserved areas. The tradeoff is read performance — since a volume's data is distributed across the aggregate rather than in reserved zones, reading requires locating scattered blocks, adding latency. NetApp introduced PAM (Performance Acceleration Module) — a PCIe-connected flash card — to cache block location metadata and frequently read data, largely eliminating this read-latency disadvantage and making WAFL's overall I/O profile competitive with traditional architectures.
Q7 A client reports they cannot create new files on a volume even though df -h shows 40% free space. What is the likely cause and how do you fix it?
The volume has exhausted its inode pool. ONTAP assigns a fixed number of inodes per volume based on volume size at creation (default: 1 inode per 32 KB). If millions of small files have been written, all inodes may be consumed while disk space remains. Verify using volume show -fields files,files-used. Resolution: if the used count equals the files count, you have two options. First, increase the inode count with volume modify -files <new_count> — up to the maximum of 1 inode per 4 KB of volume capacity. Second, grow the volume, which automatically increases the inode pool proportionally. Increasing inodes does draw from the aggregate's inode pool, so verify aggregate-level inode availability as well.
Q8 When should Continuously Available (CA) shares be used and when should they be avoided?
CA shares should be used exclusively for Hyper-V virtual machines running on ONTAP SMB3 storage and SQL Server workloads using SMB3 for database files. These workloads are sensitive to session interruption — a dropped SMB session while SQL or Hyper-V is writing can cause data corruption or VM crashes. CA shares maintain persistent session and connection IDs across brief interruptions, enabling transparent reconnection. For all other use cases (general file shares, home directories, department shares), CA is explicitly not recommended — it consumes additional controller resources to maintain persistent sessions for no meaningful benefit to regular file share workloads. ONTAP itself displays a warning when you attempt to add CA to a non-applicable share.
