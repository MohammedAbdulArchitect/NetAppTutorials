SECTION 01

Partitioned Disk Ownership

With Advanced Drive Partitioning (ADP), each physical disk is sliced into logical partitions independently assigned to different controllers. Run disk show — if Type shows shared, disks are partitioned. Use disk partition show (advanced privilege) for confirmation.

ADP v1 vs ADP v2 Partition Layouts

📀 ADP v1 — Two partitions per disk
ROOT~25%
DATA~75%
📀 ADP v2 — Three partitions per disk
ROOT~10%
DATA 1~45%
DATA 2~45%
Figure 1 — ADP partition layouts. ADP v2 (3-partition) is most common on modern AFF/FAS platforms.

Ownership Distribution Strategy

With ADP v2, a 24-drive shelf generates 24× Root, 24× Data-1, and 24× Data-2 partitions. The default balanced HA split:

Figure 2 — Default ownership distribution for ADP v2 across an HA pair. Root partitions split 12/12; data partitions go to respective controllers.
✅
Flexible Ownership
Ownership is fully flexible. If Controller-A needs more capacity, assign both Data-1 and Data-2 partitions to it — ONTAP's software-based ownership model imposes no hard constraints.

SECTION 03

Converting Non-Partitioned Disks

🚫
No Mixing in Aggregates
Partitioned (ADP) and non-partitioned disks cannot be mixed inside a single aggregate. They may exist on the same controller, but must be in separate aggregates.

To convert, enter diagnostic privilege, identify an existing partitioned disk of the same model as a reference, then run:

disk create-partition -disk <target> -reference-disk <source>

ONTAP displays the layout and prompts for confirmation. After conversion, the disk appears as partitions within the spare pool.

SECTION 04

Configuration Storage: 7-Mode vs Cluster Mode

⚠️ 7-Mode — File-Based Config

Configuration stored in plain text files under /etc/ in root volume (vol0). Every change required both a CLI command and a manual file edit — forgetting the file edit caused settings to be lost on reboot. No built-in replication to partner nodes.

✅ Cluster Mode — Database-Driven Config

All configuration managed via the Replication Database (RDB) stored inside /mroot/. A single command updates the database automatically; files are written by the database. RDB replicates to all nodes via the cluster private network.

🗄️
SECTION 05

Replication Database (RDB)

The RDB is the configuration backbone of ONTAP Cluster Mode. Every node maintains its own local RDB instance, continuously synchronized through the cluster private network. One node's RDB is elected Master — it coordinates propagation to all others. When the master reboots, another node is elected; the original node does not reclaim master role on return.

Figure 4 — RDB synchronization model. Master RDB ensures all node databases stay in sync via the cluster private interconnect.

RDB Sub-Databases

VLDB
Volume Location Database
Tracks the location and metadata of all aggregates and volumes across the cluster. Updated and replicated whenever a volume is created on any node.
VIFMGR
Virtual Interface Manager
Stores all network configuration — LIFs, port groups, VLANs, and interface failover policies. All NAS/SAN networking definitions live here.
BComDB
Block Communication & Operational Manager
Manages SAN protocol configurations: iSCSI targets, FC initiator groups, LUN mappings, and block access operational data.
MGMTD
Management Daemon
Handles cluster internal operations — node health monitoring, storage failover coordination, background processes, and event-driven automation.
CRS  MetroCluster Only
Cluster Replication Service
Active only when configured as MetroCluster. In standard clusters, CRS exists but remains dormant. Introduced in ONTAP 8.3 — this was one of the most significant architectural changes in ONTAP history, enabling synchronous replication across geographically separated sites.

Version note: ONTAP ≤8.2 has 4 RDB sub-databases. ONTAP 8.3+ adds CRS as the 5th.

SECTION 06

Cluster Quorum

The Quorum Condition

More than 50% of cluster nodes must be operational

Failing quorum causes ONTAP to forcibly shut down surviving nodes to prevent data corruption from split-brain scenarios.

Quorum Scenarios — 6-Node Cluster

✅ Scenario A: 2 nodes down, 4 nodes up (67% healthy)
QUORUM SATISFIED — Cluster operational
⚠️ Scenario B: 3 nodes down, 3 nodes up (50% healthy) — Tie condition
QUORUM INDETERMINATE — Epsilon decides
🚨 Scenario C: 4 nodes down, 2 nodes up (33% healthy)
QUORUM FAILED — ONTAP shuts down surviving nodes

SECTION 07

Epsilon — The Cluster Tiebreaker

Epsilon is a logical software token residing on exactly one node. It functions as a tie-breaking vote when exactly half the cluster nodes are down. Think of it as a casting vote in a tied parliamentary vote.

Figure 8 — Epsilon tiebreaker in action. With 2 of 4 nodes down, Epsilon on Node-1 provides the deciding vote, keeping the cluster alive.
Cluster SizeEpsilon Applicable?Reason
1-node clusterDisabledSingle node is always the master
2-node clusterDisabledTwo-node HA relies on HA mediator, not Epsilon
4-node clusterActive50/50 tie is possible (2 vs 2)
6-node and aboveActiveOdd-majority scenarios possible
ONTAP CLI — Epsilon Management
# Advanced mode required for Epsilon operations cluster::> set advanced # Check which node holds Epsilon cluster::*> cluster show Node Health Eligibility Epsilon -------------- ------- ------------ ------- storage-01 true true true storage-02 true true false storage-03 true true false storage-04 true true false # Move Epsilon: remove from current node, assign to new node cluster::*> cluster modify -node storage-01 -epsilon false cluster::*> cluster modify -node storage-03 -epsilon true
🚨
Critical Best Practice — Epsilon Before Maintenance
Never reboot a node that holds Epsilon without first relocating it. If Epsilon is on Node-1 and you reboot it, any coincidental failure elsewhere can break quorum unexpectedly.

Also, do not move Epsilon to the HA partner of the node being rebooted — it already carries extra load during takeover. Move Epsilon to a node in a different HA pair.

SECTION 08

What Is an Aggregate?

An aggregate is a logical storage pool created by grouping physical disks and applying RAID protection across them. Volumes (the logical partitions NFS/CIFS/iSCSI clients access) are created on top of aggregates. Every node ships with a root aggregate (aggr0_<nodename>) for controller configuration only — all client data goes on separately created aggregates.

Two mandatory parameters when creating an aggregate: RAID Type (protection level) and RAID Group Size (disks per stripe set — drives usable capacity and rebuild granularity).

🛡️
SECTION 09

RAID Types & Group Sizing

RAID-DP
Double Parity. 2 parity drives per RAID group. Survives 2 simultaneous failures. Industry default for FAS/AFF.
RAID-TEC
Triple Erasure Coding. 3 parity drives. Survives 3 simultaneous failures. Best for large-capacity drives (8TB+).
RAID4
Single parity. Rarely used in FAS/AFF. Only appropriate for V-Series (third-party back-end arrays).
RAID0
No parity / striping only. Zero data protection. Only used with V-Series where the back-end array provides its own RAID.
✅
Practical Guidance
Use RAID-DP for the vast majority of deployments. Move to RAID-TEC only when using very large capacity drives (8TB+) where rebuild time significantly increases the double-failure risk window.

RAID Group Sizing & Usable Capacity

ONTAP groups disks into RAID Groups (RG0, RG1, …). Each RG contains a fixed number of disks per the -maxraidsize parameter. Larger RGs = more usable capacity but longer rebuild times. SSD RAID-DP limits: min 5, default ~18, max 28 disks per RG.

SECTION 10

Creating & Managing Aggregates

ONTAP CLI — Aggregate Creation & Management
# Method 1: Auto-select disks by count, type, and size cluster::> storage aggregate create -aggregate aggr1_node01 -node storage-ctrl-01 -diskcount 12 -disktype SSD -raidtype raid_dp -maxraidsize 6 # Method 2: Specify exact disk IDs manually cluster::> storage aggregate create -aggregate aggr2_node02 -node storage-ctrl-02 -disklist 2.1.1,2.1.2,...,2.1.12 -raidtype raid_dp # Verify aggregate and RAID group layout cluster::> storage aggregate show cluster::> storage aggregate show-status -aggregate aggr1_node01 # Check aggregate status, capacity, state cluster::> storage aggregate show -fields size,availsize,usedsize,state,raidtype # Expand aggregate by adding disks cluster::> storage aggregate add-disks -aggregate aggr1_node01 -diskcount 6 -disktype SSD # Take offline and delete (volumes must be removed first) cluster::> storage aggregate offline -aggregate aggr1_node01 cluster::> storage aggregate delete -aggregate aggr1_node01
⚠️
Deletion Prerequisite
An aggregate cannot be deleted while it hosts volumes. State degraded indicates a drive failure. Always bring an aggregate offline before deletion. Deleted disks return to the spare pool.

SECTION 11

Disk Zeroing

Disks released from an aggregate return to the spare pool as not-zeroed — still containing old data. ONTAP must zero them before reuse. For hot-spare replacement, a not-zeroed spare must be zeroed first, extending the degraded window and increasing double-failure risk.

Figure 10 — Zeroing impact on aggregate creation speed and hot-spare replacement time.
ONTAP CLI — Disk Zeroing

Best Practice — Keep Spares Zeroed
Run disk zerospares proactively whenever disks return to the spare pool. This ensures immediate hot-spare replacement on disk failure, minimizing the double-failure vulnerability window.

SECTION 12

Scenario-Based Interview Q&A

❓ Q1: You need to take Node-2 offline for maintenance in a 4-node cluster. What preparation steps should you take?
Run cluster show in advanced mode to check if Node-2 holds Epsilon. If it does, relocate Epsilon to Node-3 or Node-4 (never Node-1, its HA partner). Verify the HA partner can take over via storage failover show and confirm volumes are accessible via alternative paths. Without relocating Epsilon, if Node-2 goes down holding it, ONTAP treats it as if a critical vote was lost — any coincidental failure during maintenance could break quorum.
❓ Q2: What is the RDB and how does it differ from 7-Mode configuration management?
The RDB is a distributed database system in ONTAP Cluster Mode storing all cluster configuration state. Unlike 7-Mode, where you had to both run a command AND manually edit files under /etc/, Cluster Mode automatically writes configuration changes to the underlying files via the RDB and replicates them to all nodes. ONTAP 8.2 had 4 RDB sub-units; from 8.3 onwards a 5th (CRS) was added for MetroCluster support.
❓ Q3: In a 6-node cluster, 3 nodes fail simultaneously. Will the cluster survive?
With exactly 3 of 6 nodes down (50% failure), the cluster enters a tie condition. Epsilon is decisive: if it is on a surviving node, ONTAP counts it as an additional fractional vote, keeping the cluster operational. If Epsilon is on a failed node, quorum fails and ONTAP shuts down all remaining nodes to prevent split-brain. The outcome is entirely determined by Epsilon's placement.
❓ Q4: Designing an aggregate with 18 SSD disks and RAID-DP, RAID group size 9 — what is usable capacity?
Two RAID groups of 9 disks each. Each group uses 2 parity disks (P + DP), leaving 7 data disks per group — 14 data disks total. With 4TB disks, usable capacity is approximately 56TB. Using RAID group size 18 (single group) gives 16 data disks (~64TB usable) but increases rebuild time. Always distribute disks evenly across RAID groups.
❓ Q5: What is the significance of the "not-zeroed" state and when is it a serious risk?
A not-zeroed spare must be formatted before ONTAP can use it. The critical risk is during hot-spare rebuild: if the spare is not-zeroed, ONTAP zeroes it before starting RAID reconstruction. During that zeroing window, the aggregate is degraded with no spare. If a second disk fails before zeroing completes, the aggregate may be irrecoverable. Mitigation: always run disk zerospares proactively.
❓ Q6: Why can't partitioned (ADP) and non-partitioned disks be mixed in the same aggregate?
ADP disks present logical partition units (Data-1, Data-2, Root) rather than whole physical disks. The capacity sizing, block addressing, and RAID stripe calculations differ fundamentally from whole-disk drives. ONTAP's RAID layer cannot reconcile these within a single RAID group — consistent parity calculations would be impossible, introducing data integrity risks. Both types can coexist on the same controller in separate aggregates.
