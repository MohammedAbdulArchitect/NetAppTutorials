SECTION 01

Partitioned Disk Ownership

When ONTAP implements Advanced Drive Partitioning (ADP), each physical disk is sliced into logical partitions that can be independently assigned to different controllers. Unlike traditional whole-disk ownership, partitioned disks require a more granular ownership model.

ℹ️
Identifying Partitioned Disks
Run disk show — if the Type column shows shared, those disks are partitioned. Non-partitioned disks display a standard type like SAS or SSD. For definitive confirmation, use disk partition show at advanced privilege.

ADP v1 vs ADP v2 Partition Layouts

📀 ADP v1 — Two partitions per disk
ROOT~25%
DATA~75%
📀 ADP v2 — Three partitions per disk
ROOT~10%
DATA 1~45%
DATA 2~45%
Figure 1 — ADP partition layouts. ADP v2 (3-partition) is most common on modern AFF/FAS platforms.

Ownership Distribution Strategy

With ADP v2, a shelf of 24 drives generates three distinct partition types. The recommended distribution in an HA pair (Controller A + Controller B) follows a balanced split:

Figure 2 — Default ownership distribution for ADP v2 across an HA pair. Root partitions split 12/12; data partitions go to respective controllers.
✅
Flexible Ownership
Partition ownership is fully flexible. If Controller-A needs more storage capacity, you can assign both Data-1 and Data-2 partitions to it. ONTAP's software-based ownership model imposes no hard constraints — design it to match your workload requirements.

The -partition-type option in disk remove-ownership and disk assign is only visible at advanced or diagnostic privilege levels. Attempting these commands in standard admin mode will not show partition-specific parameters.

SECTION 03

Converting Non-Partitioned Disks to Partitioned

In some environments, a mix of non-partitioned and partitioned disks may exist on the same controller. While both can coexist, they cannot be combined within a single aggregate. If uniformity is required, ONTAP allows converting non-partitioned disks by using an existing partitioned disk as a sizing reference.

🚫
No Mixing in Aggregates
Partitioned (ADP) and non-partitioned disks cannot be mixed inside a single aggregate. They may exist on the same controller, but must be in separate aggregates. Attempting to mix them is unsupported and will result in an error.
1
Enter Diagnostic Mode
The disk create-partition command is only accessible at the diagnostic privilege level.
2
Identify a Reference Disk
Select an existing partitioned disk of the same model/size as your reference (source). ONTAP will replicate its partition sizing onto the target disk.
3
Execute the Conversion
Run disk create-partition -disk <target> -reference-disk <source>. ONTAP will display the partition layout and prompt for confirmation before proceeding.
4
Verify in Spare Pool
After conversion, the disk transitions from displaying separately to appearing as partitions within the spare pool, confirming successful conversion.

SECTION 04

Configuration Storage: 7-Mode vs Cluster Mode

Understanding how ONTAP stores controller configurations helps explain both the architectural shift from legacy 7-Mode to Cluster Mode, and why the Replication Database (RDB) is central to modern ONTAP reliability.

⚠️ 7-Mode — File-Based Config

Configuration is stored in plain text files under /etc/ in the root volume (vol0). Files like rc, hosts, and resolv.conf must be manually edited.

  • Every change requires two steps: run a command and edit the config file
  • Forgetting to update the file causes settings to be lost on reboot
  • Direct file edits carry risk of corrupting other settings
  • No built-in replication of config state to partner nodes
✅ Cluster Mode — Database-Driven Config

All configuration is managed via the Replication Database (RDB) stored inside /mroot/ of each node's root volume. Files are written automatically by the database — no manual editing required.

  • Single command updates the database; file update is automatic
  • Configurations persist across reboots without extra steps
  • RDB replicates config to all nodes via cluster private network
  • Eliminates the risk of inconsistent file-level edits
🗄️
SECTION 05

Replication Database (RDB)

The Replication Database (RDB) is the configuration backbone of ONTAP Cluster Mode. Every node in a cluster maintains its own local RDB instance, and all instances are continuously synchronized through the cluster private network — ensuring configuration consistency across the entire cluster.

Figure 4 — RDB synchronization model. Master RDB ensures all node databases stay in sync via the cluster private interconnect.

RDB Sub-Databases

The RDB is organized into specialized sub-databases, each responsible for a distinct aspect of cluster configuration:

VLDB
Volume Location Database
Tracks the location and metadata of all aggregates and volumes across the cluster. When a volume is created on any node, VLDB is updated and the information replicated to every other node's VLDB instance.
VIFMGR
Virtual Interface Manager
Stores all network configuration state — logical interfaces (LIFs), port groups, VLANs, and interface failover policies. All NAS/SAN networking definitions live here.
BComDB
Block Communication & Operational Manager
Manages all SAN protocol configurations including iSCSI targets, FC initiator groups, LUN mappings, and zoning-related operational data for block access protocols.
MGMTD
Management Daemon
Handles cluster internal operations — node health monitoring, storage failover coordination, background processes, and event-driven automation. If failover doesn't behave as expected, MGMTD may be the place to investigate.
CRS  MetroCluster Only
Cluster Replication Service / Session
An optional sub-database that becomes active only when the cluster is configured as a MetroCluster. In standard (non-Metro) clusters, CRS exists but remains dormant. Introduced in Data ONTAP 8.3, this was one of the most significant architectural changes in ONTAP history — enabling synchronous replication across geographically separated sites.
💡
Interview Tip — RDB Version History
Up to Data ONTAP 8.2: RDB has 4 sub-databases (no CRS, as MetroCluster wasn't supported in cluster mode). From Data ONTAP 8.3 onwards: RDB gains a 5th sub-database (CRS) as MetroCluster support was introduced. The 8.2→8.3 upgrade was historically one of the most complex ONTAP upgrades ever performed.

Master vs Secondary RDB — Election Process

Within a cluster, one node's RDB is elected as the Master RDB. The master carries additional responsibility — it ensures configuration changes are propagated to all other nodes. There is no privilege or special access associated with being master; it is purely an internal coordination role.

SECTION 06

Cluster Quorum

Quorum is a cluster health condition that ONTAP evaluates continuously. For a cluster to remain in a functional, data-serving state, it must satisfy the quorum condition at all times. Failing to satisfy quorum causes ONTAP to forcibly shut down surviving nodes to prevent data corruption from split-brain scenarios.

The Quorum Condition

More than 50% of cluster nodes must be operational

A majority of nodes must be healthy and reachable for the cluster to serve data.

Quorum Scenarios — 6-Node Cluster

✅ Scenario A: 2 nodes down, 4 nodes up (67% healthy)
QUORUM SATISFIED — Cluster operational
⚠️ Scenario B: 3 nodes down, 3 nodes up (50% healthy) — Tie condition
QUORUM INDETERMINATE — Epsilon decides
🚨 Scenario C: 4 nodes down, 2 nodes up (33% healthy)
QUORUM FAILED — ONTAP shuts down surviving nodes
Figure 6 — Quorum scenarios for a 6-node cluster. Only 50/50 ties require Epsilon intervention.
ℹ️
Cluster Private Network — The Lifeline
Even when an entire HA pair goes offline, the cluster private network allows remaining active nodes to access the storage connected to those failed nodes — serving data through indirect paths. Performance may degrade, but data remains accessible as long as quorum is satisfied. This is the key purpose of the high-speed cluster interconnect.

SECTION 07

Epsilon — The Cluster Tiebreaker

Epsilon is a logical software token that resides on exactly one node in a multi-node cluster. It functions as a tie-breaking vote when exactly half the cluster nodes are down. Think of it like a casting vote in a tied parliamentary vote — whoever holds it determines the outcome.

Figure 8 — Epsilon tiebreaker in action. With 2 of 4 nodes down, Epsilon on Node-1 provides the deciding vote, keeping the cluster alive.

Epsilon Applicability

Cluster SizeEpsilon Applicable?Reason
1-node clusterDisabledNo quorum concept — single node is always the master
2-node clusterDisabledTwo-node HA relies on HA mediator, not Epsilon
4-node clusterActive50/50 tie is possible (2 vs 2); Epsilon resolves it
6-node and aboveActiveOdd-majority scenarios possible; Epsilon provides tiebreaking

Managing Epsilon

ONTAP CLI — Epsilon Management
# Advanced mode required for Epsilon operations cluster::> set advanced # Check which node holds Epsilon (true = Epsilon is here) cluster::*> cluster show Node Health Eligibility Epsilon -------------- ------- ------------ ------- storage-01 true true true storage-02 true true false storage-03 true true false storage-04 true true false # Move Epsilon: first remove from current node cluster::*> cluster modify -node storage-01 -epsilon false # Then assign Epsilon to a different node cluster::*> cluster modify -node storage-03 -epsilon true
🚨
Critical Best Practice — Epsilon Before Maintenance
Never reboot a node that holds Epsilon without first relocating it. If Epsilon is on Node-1 and you reboot Node-1, ONTAP treats this as if a node holding a critical vote has failed. If coincidentally another node in the cluster also goes down during that window, your cluster can unexpectedly lose quorum.

Additionally, when moving Epsilon away from a node you plan to reboot, do not move it to the HA partner of that node. The HA partner will absorb the rebooting node's workload and already carries extra load — adding Epsilon on top increases the risk of cascading failure. Move Epsilon to a node in a different HA pair.

SECTION 08

What Is an Aggregate?

An aggregate is a logical storage pool created by grouping physical disk drives and applying RAID protection across them. It forms the foundational storage layer in ONTAP — volumes (logical partitions that NFS/CIFS/iSCSI clients access) are created on top of aggregates.

🔑 Two Mandatory Parameters When Creating an Aggregate
1. RAID Type

Defines the protection level. Determines how many parity drives are used and how much redundancy you get.

2. RAID Group Size

Defines how many disks are grouped together into a single RAID stripe set (RAID group / RG). This drives usable capacity and fault tolerance granularity.

ℹ️
Default and Root Aggregates
Every ONTAP node ships with a root aggregate named aggr0_<nodename>. This aggregate hosts only controller configuration data — it is never used for client data. If you have a 6-node cluster, you'll have 6 root aggregates (one per node). All user-facing volumes must be created on additional aggregates you create.
🛡️
SECTION 09

RAID Types in ONTAP

ONTAP supports multiple RAID variants. Each offers a different balance of protection, disk overhead, and suitability for specific storage tiers.

RAID-DP
Double Parity. 2 dedicated parity drives per RAID group. Can survive 2 simultaneous disk failures. Industry default for FAS/AFF.
RAID-TEC
Triple Erasure Coding. 3 parity drives. Survives 3 simultaneous failures. Best for large-capacity drives (8TB+) where reconstruction takes longer.
RAID4
Single parity. 1 parity drive per RAID group. Rarely used in FAS/AFF. Only appropriate for V-Series (third-party back-end arrays).
RAID0
No parity / striping only. Zero data protection. Only used with V-Series where the back-end array already provides its own RAID protection.
✅
Practical Guidance
For the vast majority of deployments: use RAID-DP. Only move to RAID-TEC if you're using very large capacity drives (8TB+, 10TB+ SSDs) where a rebuild after a disk failure takes a long time, increasing the window of vulnerability to a second failure.

RAID Group Sizing & Usable Capacity

When creating an aggregate, ONTAP groups your disks into one or more RAID Groups (RG), named RG0, RG1, RG2, etc. Each RAID group contains a fixed number of disks as defined by the RAID group size parameter.

Larger RAID Groups = More Usable Capacity
A larger RAID group size means fewer parity disks relative to your total disk count, resulting in more usable space. However, it also increases rebuild time if a disk fails. SSD RAID-DP limits: minimum 5, default ~18, maximum 28 disks per RAID group. Best practice: distribute disks evenly across RAID groups.

SECTION 10

Creating Aggregates

ONTAP provides a flexible aggregate creation workflow that lets you either specify disk IDs manually or rely on ONTAP intelligence to select optimal drives automatically. The system always previews the layout before committing.

ONTAP CLI — Aggregate Creation
# Method 1: Auto-select disks by count, type, and size cluster::> storage aggregate create -aggregate aggr1_node01 -node storage-ctrl-01 -diskcount 12 -disktype SSD -raidtype raid_dp -maxraidsize 6 # ONTAP previews the layout — confirm with 'yes' # Method 2: Specify exact disk IDs manually cluster::> storage aggregate create -aggregate aggr2_node02 -node storage-ctrl-02 -disklist 2.1.1,2.1.2,2.1.3,2.1.4,2.1.5,2.1.6,2.1.7,2.1.8,2.1.9,2.1.10,2.1.11,2.1.12 -raidtype raid_dp # Verify aggregate after creation cluster::> storage aggregate show # Check detailed RAID group layout cluster::> storage aggregate show-status -aggregate aggr1_node01

Managing Existing Aggregates

ONTAP CLI — Aggregate Management
# Check aggregate status and capacity cluster::> storage aggregate show -fields size,availsize,usedsize,state,raidtype # Take an aggregate offline before deletion cluster::> storage aggregate offline -aggregate aggr1_node01 # Delete aggregate (volumes must be removed first) cluster::> storage aggregate delete -aggregate aggr1_node01 # Add disks to expand existing aggregate cluster::> storage aggregate add-disks -aggregate aggr1_node01 -diskcount 6 -disktype SSD
⚠️
Deletion Prerequisite
An aggregate cannot be deleted while it hosts volumes. The state shown as degraded indicates a drive failure within that aggregate. Always bring an aggregate offline before attempting deletion. After deletion, previously used disks return to the spare pool.

SECTION 11

Disk Zeroing — Why It Matters

When disks are released from an aggregate, they return to the spare pool in a not-zeroed state. This means they still contain old data blocks. ONTAP must zero (format/wipe) these disks before incorporating them into a new aggregate.

Proactively run disk zerospares whenever disks return to the spare pool. This ensures hot-spare replacements happen immediately when a drive fails — minimizing the double-failure vulnerability window during RAID rebuild.

SECTION 12

Scenario-Based Interview Questions & Answers

❓ Q1: You have a 4-node ONTAP cluster and need to take Node-2 offline for hardware maintenance. What preparation steps should you take before shutting it down?
First, run cluster show in advanced mode to check if Node-2 holds Epsilon. If it does, relocate Epsilon to Node-3 or Node-4 (never Node-1, which is its HA partner). Verify the HA partner can take over Node-2's workloads via storage failover show. Confirm all volumes are accessible via alternative paths. Only then proceed with the shutdown. Without relocating Epsilon, if Node-2 goes down while Epsilon is on it, the cluster believes it has lost 2 votes instead of 1, endangering quorum.
❓ Q2: What is the RDB, and how is it different from configuration file management in 7-Mode?
The Replication Database (RDB) is a distributed, multi-node database system in ONTAP Cluster Mode that stores all cluster configuration state. Unlike 7-Mode, where an administrator had to both run a command AND manually edit files under /etc/, Cluster Mode automatically writes configuration changes to the underlying files via the RDB. The RDB also replicates configuration across all nodes via the cluster private network, ensuring consistency. ONTAP 8.2 had 4 RDB sub-units; from 8.3 onwards a 5th (CRS) was added for MetroCluster support.
❓ Q3: In a 6-node cluster, 3 nodes fail simultaneously. Will the cluster survive? What determines the outcome?
With exactly 3 of 6 nodes down (50% failure), the cluster enters a tie condition — it cannot satisfy the simple majority quorum rule on its own. This is precisely where Epsilon comes into play. If Epsilon is hosted on one of the 3 surviving nodes, ONTAP counts it as an additional fractional vote, tipping the balance toward the surviving side and keeping the cluster operational. If Epsilon is on one of the failed nodes, quorum fails and ONTAP shuts down all remaining nodes to prevent a split-brain. The outcome is entirely determined by Epsilon's placement.
❓ Q4: You're designing an aggregate using 18 SSD disks with RAID-DP. What RAID group size would you use, and how does it affect usable capacity?
For 18 SSDs with RAID-DP and a RAID group size of 9: ONTAP creates 2 RAID groups of 9 disks each. Each group uses 2 disks for parity (P + DP), leaving 7 data disks per group — totaling 14 data disks. If each disk is 4TB, usable capacity is approximately 56TB. Using a larger RG (e.g., 18 as a single group) would give 16 data disks (~64TB usable) but increases reconstruction time on failure. Using smaller RGs (e.g., 6×3 groups) gives less usable space. The default ONTAP RG size for SSD RAID-DP is typically around 18, and maximum is 28 — always distribute disks evenly across RAID groups.
❓ Q5: What is the significance of the "not-zeroed" disk state, and when does it become a serious operational risk?
A not-zeroed spare disk still contains residual data from a previous aggregate and must be formatted before ONTAP can use it. The risk manifests during a hot-spare rebuild: if a disk fails in an aggregate and the designated hot-spare is not-zeroed, ONTAP must first zero it before starting the RAID rebuild. During this zeroing period, the aggregate is in a degraded state with no spare. If a second disk fails before zeroing completes and the new spare is ready, the aggregate may be irrecoverable. The mitigation is always running disk zerospares proactively to keep the spare pool in a ready-to-use state.
❓ Q6: Why can't you mix partitioned (ADP) and non-partitioned disks in the same aggregate?
Partitioned disks in ADP present logical partition units (Data-1, Data-2, Root) to ONTAP rather than whole physical disks. Non-partitioned disks present a full physical disk. The capacity sizing, block addressing, and RAID stripe calculations differ fundamentally between the two disk types. ONTAP's RAID layer cannot reconcile these differences within a single RAID group — mixing them would make consistent parity calculations impossible and introduce data integrity risks. Both types can coexist on the same controller, but they must be in separate aggregates.

SECTION 13

Summary & Key Takeaways

🔑 Partition Ownership
  • ADP v2 creates 3 partitions: Root, Data-1, Data-2
  • Data-1 → Controller-A; Data-2 → Controller-B (default balanced split)
  • Root partitions split 50/50 between HA pair nodes
  • Auto-assign is the recommended default — disable only for custom layouts
  • Partition commands require advanced or diagnostic privilege
🗄️ RDB Architecture
  • RDB replaces 7-Mode's manual config file model
  • 5 sub-databases: VLDB, VIFMGR, BComDB, MGMTD, CRS
  • CRS appeared in ONTAP 8.3 for MetroCluster support
  • Master RDB ensures all nodes stay in sync
  • Master role migrates on reboot — never returns to original node
⚖️ Quorum & Epsilon
  • Quorum requires >50% of nodes operational
  • Epsilon is a tiebreaker for exact 50/50 split scenarios
  • Epsilon only applies to clusters of 4+ nodes
  • Never reboot the Epsilon-holding node without first relocating it
  • Move Epsilon to a node in a different HA pair during maintenance
🧱 Aggregates
  • Aggregate = logical pool of physical disks + RAID protection
  • RAID-DP is the standard for 90%+ of deployments
  • Larger RAID groups = more usable capacity, longer rebuild time
  • Keep spare disks zeroed to minimize hot-spare replacement delays
  • Volumes (client data) live on top of aggregates, not directly on disks
