SECTION 01

ONTAP CLI Shells

ONTAP provides three shell environments. Shells are separate entry points into different administrative layers — distinct from modes, which are privilege levels within the Cluster Shell.

🟢 Cluster Shell

Primary environment for all day-to-day operations. Cluster-wide scope — where 95% of your work happens.

Default Entry Point Cluster-Scoped
🔵 Node Shell

Legacy 7-Mode compatibility layer. Node-specific scope. Certain disk-level commands still require a visit here. No tab completion.

Node-Scoped Limited Commands
🔴 System Shell

Linux-based OS layer for deep diagnostic log collection only. No configuration changes. Invoked under vendor support guidance.

Node-Specific Diagnostics Only

SECTION 02

Privilege Modes in ONTAP

Within the Cluster Shell, three privilege modes control which commands are accessible. Each level unlocks additional capabilities while increasing configuration risk.

Admin Mode
ontap-cluster::>
Default on login. Handles ~80% of routine tasks — volumes, LIFs, health checks, statistics.
Default
Advanced Mode
ontap-cluster::*>
Unlocks additional parameters and tuning options. The ★ in the prompt confirms elevated privileges.
★ Prompt
Diagnostic Mode
ontap-cluster::*>
Most privileged. Can affect cluster stability. Only use under vendor-guided troubleshooting.
Expert Only
ONTAP CLI — Mode Navigation
# Enter Advanced mode ontap-cluster::> set advanced ontap-cluster::*> # Star (*) confirms advanced/diag mode # Enter Diagnostic mode ontap-cluster::*> set diag # Return to Admin mode ontap-cluster::*> set admin ontap-cluster::> # Star disappears # Shortcuts set adm # → Admin set adv # → Advanced set diag # → Diagnostic
🚨
Diagnostic Mode Risk
Commands in diagnostic mode can disrupt data access or corrupt configuration. Never enter without a specific known reason. Always return to admin mode when done — the ★ in the prompt is your reminder.

SECTION 04

IOM Modules & ACP

Every SAS disk shelf contains exactly two IOM (Input/Output Module) modules — IOM-A and IOM-B — operating simultaneously in active-active. If one IOM or SAS path fails, the other continues without interruption. The IOM model determines the shelf's maximum throughput.

3 Gb/s
IOM3
DS40-43 series
6 Gb/s
IOM6
DS2246, DS4246 series
12 Gb/s
IOM12
DS212C, DS224C series
⚠️
Never Mix IOM Types in a Single Loop
Mixing IOM generations on one daisy-chain loop degrades the entire loop to the slowest IOM's speed. Always create separate SAS loops per IOM generation.

ACP — Alternate Control Path

On older shelves, shelf health monitoring and data traffic use separate connections. ACP is a dedicated Ethernet management path (via controller ports e0P / e0I). On current shelves (DS212C, DS224C+), ACP monitoring is embedded within the SAS connection — no separate cabling needed.

🔴 Older Shelves — ACP Required
  • DS2246, DS4243 and legacy models
  • Separate Ethernet to e0P/e0I on controllers
  • SAS = data; Ethernet = monitoring
  • Must configure ACP for shelf health alerts
🟢 Newer Shelves — ACP Embedded
  • DS212C, DS224C, DS460C and current models
  • ACP monitoring embedded within SAS protocol
  • Single SAS cable carries data + management
  • No separate ACP configuration required
🔄
SECTION 05

Daisy-Chain (Multi-Path) Shelf Topology

ONTAP uses a daisy-chain topology where controller SAS ports connect to the first shelf, each subsequent shelf chains port-to-port, and the last shelf loops back to the second controller — creating a single redundant loop reachable by both controllers simultaneously.

Fig 5.1 — Dual-Path Daisy-Chain SAS Shelf Topology with IOM-A and IOM-B Loops
✅
Adding Shelves — Non-Disruptive
Disconnect the last shelf's outbound SAS port from the controller, insert the new shelf, reconnect — one path at a time to maintain continuous data access throughout.

SECTION 06

Setting Shelf IDs

Each shelf must have a unique numeric ID before connecting to a controller. This ID becomes part of every disk's permanent address and is essential for fault isolation.

  • 1Locate the LCD display panel and button on the front or back of the shelf.
  • 2Press and hold the button — the display cycles through 0–32.
  • 3Release at your desired number. The shelf ID is set immediately — no ONTAP command needed.
  • 4IDs must be unique within the same SAS loop. IDs can be reused across separate loops.
⚠️
Shelf ID Changes Are Disruptive in Production
Once a shelf is in service with live data, changing its shelf ID requires taking the shelf offline. Plan shelf IDs carefully before deployment — changes are free only on shelves with no assigned disks.

SECTION 08

Disk Ownership in ONTAP

Both controllers in an HA pair see all attached disks, but each disk can only be owned by one controller at a time. Ownership determines which controller manages the disk and can use it in an aggregate. Unassigned disks sit in an effective spare pool visible to both controllers.

SECTION 09

Advanced Disk Partitioning (ADP)

ADP logically divides a physical drive into separate root and data partitions. Without ADP, ONTAP reserves entire disks for root aggregates — wasting significant capacity, especially on expensive flash drives.

✂️ ADP v1 — FAS Mid-Range
  • 2 partitions per drive: 1 root + 1 data
  • Applied only to shelves at initial cluster configuration
  • ADP and non-ADP disks cannot mix in the same aggregate
  • Optional for FAS 8000/9000; factory-set for FAS 2500/2600/2700
  • Conversion requires a full cluster reset — destructive
✂️ ADP v2 — AFF All-Flash
  • 3 partitions per drive: 1 root + 2 equal data
  • Mandatory on all AFF systems — cannot be disabled
  • All shelves including future expansions are automatically ADP v2
  • Data partitions equal in size — capacity split evenly
  • Enables flexible aggregate design with independent partition assignment

SECTION 10

Shelf & Disk Firmware Management

In ONTAP 9+, shelf and disk firmware are bundled inside the ONTAP image and upgraded automatically — non-disruptively — during an ONTAP upgrade. Individual firmware updates are now the exception, not the rule.

🕒 Legacy (Pre-ONTAP 9)
  • ONTAP OS, shelf firmware, and disk firmware each required separate manual upgrade processes
  • Individual planning, scheduling, and validation for each
✅ Modern (ONTAP 9+)
  • Shelf and disk firmware bundled inside ONTAP image
  • All components upgraded automatically — background, non-disruptive
  • Individual firmware update only needed for specific out-of-cycle patches

Quick Quiz

1. What symbol appears in the ONTAP CLI prompt when you are in Advanced or Diagnostic mode?
✓ Correct! The asterisk (*) in the prompt is your visual indicator that elevated privileges are active. Always return to admin mode with set adm when finished.
✗ Incorrect. The asterisk (*) appears in the prompt — e.g., ontap-cluster::*> — whenever you are in Advanced or Diagnostic mode.
2. You have a disk identified as 3.5.17. Which shelf do you walk to?
✓ Correct! Disk ID format is Channel.ShelfID.Slot — so 3.5.17 = channel 3, shelf-ID 5, slot 17.
✗ Incorrect. The format is Channel.ShelfID.Slot. So 3.5.17 = SAS channel 3, shelf with ID 5, disk slot 17.
3. On an AFF system, how many partitions does ADP create per physical drive?
✓ Correct! ADP v2 (mandatory on AFF) creates 3 partitions: 1 root + 2 equal data partitions per drive.
✗ Incorrect. AFF uses ADP v2 which creates 3 partitions per drive: 1 small root partition and 2 equal data partitions.
4. You are adding two DS224C shelves (IOM12) and one DS2246 (IOM6) to an existing cluster. How should they be connected?
✓ Correct! Different IOM generations must never be mixed on a single SAS loop — the whole loop would run at the slowest speed. Separate loops are required.
✗ Incorrect. Mixing IOM types on one loop degrades performance for all shelves. DS224C (IOM12) and DS2246 (IOM6) must be on separate SAS loops.

SECTION 12

Scenario-Based Interview Q&A

Q1 You need to run a command not available in admin mode. What do you do?
Enter advanced mode with set adv. The prompt shows * confirming elevated privileges. After completing the task, return with set adm. Never leave an interactive session in advanced or diagnostic mode unattended.
Q2 Your team purchased 2× DS224C (IOM12) and 1× DS2246 (IOM6) for expansion. How do you connect them?
Never mix IOM generations on a single loop. Connect the two DS224C shelves in one new SAS loop and the DS2246 in a separate loop, each using dedicated SAS ports on the controller (or via a SAS HBA if native ports are exhausted).
Q3 An alert shows failed disk 2.3.15. How do you physically locate it?
Decode: Channel 2 (second SAS loop group), Shelf ID 3, slot 15 (16th physical slot from zero). Walk to channel 2's cable grouping, find the shelf labeled "3" on its LCD, count to slot 15. The disk LED should blink red to confirm.
Q4 What is ADP and why is it mandatory on AFF systems?
ADP logically splits physical drives into root and data partitions, so the root aggregate consumes only ~400GB rather than entire drives. On AFF — with expensive NVMe SSDs — wasting multiple full drives for root configuration would be unacceptable. ADP v2 (the AFF version) creates three partitions per drive (1 root + 2 data), ensuring near-zero overhead from root aggregates.
Q5 Can ADP and non-ADP drives coexist in the same data aggregate?
No. ADP and non-ADP disks must be in separate aggregates. The only path to uniformity is requesting factory ADP configuration for all shelves, or a full destructive cluster reset — not possible on a system with live data.
Q6 How do you verify ADP status and version on a running cluster?
Enter diagnostic mode (set diag) and run disk partition show. Empty output = no ADP. Two partitions per disk (P1, P2) = ADP v1 (FAS). Three partitions (P1, P2, P3) = ADP v2 (AFF). Cross-reference partition sizes — the root partition will be significantly smaller.
