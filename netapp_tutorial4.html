SECTION 00

What This Tutorial Covers

Before you can effectively administer an ONTAP-based storage environment, you need a solid understanding of three foundational areas: how the command-line interface is structured and what privileges each mode grants, how physical disk shelves are connected to controllers and identified, and how ONTAP assigns and manages disk ownership — including the modern Advanced Disk Partitioning feature that maximizes capacity utilization.

This tutorial bridges those concepts end-to-end, moving from CLI navigation through physical shelf topology all the way to ADP partition management.

SECTION 01

ONTAP CLI Shells

ONTAP provides multiple command-line shell environments, each designed for a specific layer of administrative access. It is critical to understand the distinction between shells and modes — they are entirely different concepts that operate at different layers of the platform.

🟢 Cluster Shell

The primary administrative environment for all day-to-day operations. Cluster-wide commands executed here affect all nodes in the HA pair. This is where 95% of your work happens — aggregate creation, volume management, network configuration, and monitoring.

Default Entry Point Cluster-Scoped
🔵 Node Shell

A legacy compatibility layer retained from 7-Mode. Access is node-specific — commands apply only to the targeted node. Certain advanced operations (e.g., specific storage and disk-level commands) still require a brief visit here. Tab completion is not available in node shell.

Node-Scoped Limited Commands
🔴 System Shell

A Linux-based shell underneath ONTAP that grants access to the OS-level internals of a specific node. Used exclusively for deep diagnostic log collection. No configuration changes are performed here. This is typically invoked only under guidance from vendor support.

Node-Specific Diagnostics Only
⚠️
Do Not Confuse Shells with Modes
Shells are separate entry points into different administrative layers of ONTAP. Modes (admin, advanced, diagnostic) are privilege levels within the Cluster Shell. Switching modes does not change your shell context — you remain in the cluster shell and simply unlock more commands.

SECTION 02

Privilege Modes in ONTAP

Within the Cluster Shell, ONTAP implements a three-tier privilege model that controls which commands and parameters are accessible. Think of it as a graduated security model — each level unlocks additional capabilities while increasing the potential for configuration impact.

Admin Mode
ontap-cluster::>
The default mode upon login. Approximately 80% of routine operational tasks are handled here — creating volumes, managing LIFs, running health checks, and viewing statistics.
Default
Advanced Mode
ontap-cluster::*>
Unlocks additional parameters, tuning options, and commands not exposed in admin mode. The ★ symbol appears in the prompt as a visual reminder that expanded privileges are active.
★ Prompt
Diagnostic Mode
ontap-cluster::*>
The most privileged mode. Commands here can modify deep system behaviors and affect cluster stability. Only used during vendor-guided troubleshooting or for specific known procedures.
Expert Only

Navigating Between Modes

Switching between modes is straightforward and non-destructive. Any mode can be entered from any other mode directly — there is no required sequence. The prompt changes to reflect your current privilege level.

ONTAP CLI — Mode Navigation
# Check current privilege level (from any mode) ontap-cluster::> set <TAB> Current privilege level: admin # Enter Advanced mode ontap-cluster::> set advanced Warning: These advanced commands are potentially dangerous... Do you want to continue? {y|n}: y ontap-cluster::*> # Star (*) confirms advanced/diag mode # Enter Diagnostic mode (from any mode) ontap-cluster::*> set diag # Return to Admin mode ontap-cluster::*> set admin ontap-cluster::> # Star disappears — you're back in admin # Shortcut commands (faster alternative) set adm # → Admin mode set adv # → Advanced mode set diag # → Diagnostic mode (also: set d) # Verify ONTAP version ontap-cluster::> version NetApp Release 9.12.1: Tue Jul 11 ... # Clear screen ontap-cluster::> Ctrl+L
🚨
Diagnostic Mode Risk Warning
Commands available in diagnostic mode can impact cluster stability, disrupt data access, or corrupt configuration if misused. Never enter diagnostic mode without a specific known reason. If in doubt, engage vendor support before proceeding.
💡
Pro Tip — Visual Mode Indicator
The presence of a * (asterisk) in your CLI prompt means you are in either Advanced or Diagnostic mode. Always use set <TAB> to confirm exactly which elevated mode you are in before running unfamiliar commands.

Each technology determines what disk types are supported, the available bandwidth, and the shelf models that can be used. Understanding this is essential before planning a storage expansion.

SECTION 04

IOM Modules — Input/Output Architecture

Every SAS disk shelf contains exactly two IOM (Input/Output Module) modules — designated IOM-A and IOM-B. These modules are the intelligence of the shelf: they host the SAS ports used for connectivity, manage internal disk communication, and provide the physical path between controllers and drives.

Both IOM modules operate simultaneously in an active-active configuration. If one IOM or one SAS path fails, the other continues serving data without interruption — this is the foundation of shelf-level redundancy. The IOM model determines the shelf's maximum throughput speed.

3 Gb/s
IOM3
Compatible Shelf: DS40-43 series
6 Gb/s
IOM6
Compatible Shelf: DS2246, DS4246 series
12 Gb/s
IOM12
Compatible Shelf: DS212C, DS224C series
⚠️
Critical: Never Mix IOM Types in a Single Loop
If you connect shelves with different IOM generations into the same daisy-chain loop, the entire loop will operate at the lowest IOM speed — severely degrading performance of faster shelves. Always create separate SAS loops per IOM generation.

ACP — Alternate Control Path (Legacy Models)

On older disk shelf models (e.g., DS2246, DS4243), physical monitoring traffic and data traffic use separate connections. The Alternate Control Path (ACP) is a dedicated Ethernet-based management path used to monitor shelf health: temperature, disk status, power supply, and fan readings.

Controllers in these older designs expose dedicated private Ethernet ports — typically labeled e0P and e0I — specifically for ACP connections. On current-generation shelves (DS212C, DS224C and newer), ACP monitoring is bundled over the same SAS connection, eliminating the need for separate Ethernet cabling.

🔴 Older Shelves — ACP Required
  • DS2246, DS4243 and similar legacy models
  • Separate Ethernet connections to e0P/e0I ports on controllers
  • SAS handles data; Ethernet handles monitoring
  • Must configure ACP to enable shelf health alerts
🟢 Newer Shelves — ACP Embedded
  • DS212C, DS224C, DS460C and current models
  • ACP monitoring embedded within SAS protocol
  • Single SAS cable carries both data and management traffic
  • No separate ACP configuration required

SECTION 05

Daisy-Chain (Multi-Path) Shelf Topology

Rather than connecting every shelf directly to the controller (which would require an impractical number of SAS ports), ONTAP uses a daisy-chain topology. In this design, controller SAS ports connect to the first shelf, and each subsequent shelf connects port-to-port in a chain — with the last shelf looping back to the second controller.

This creates a single redundant loop where all shelves are reachable by both controllers simultaneously — ensuring zero downtime access even when individual SAS links or IOM modules fail.

Fig 5.1 — Dual-Path Daisy-Chain SAS Shelf Topology with IOM-A and IOM-B Loops
✅
Expanding Storage — Online, No Downtime
Adding a new shelf to an existing daisy-chain is a non-disruptive operation. Disconnect the last shelf's outbound SAS port from the controller, insert the new shelf into the chain, and reconnect — one path at a time to maintain continuous access throughout the procedure.

SECTION 06

Setting Shelf IDs

Each disk shelf must be assigned a unique numeric ID before it is connected to a controller. This ID forms part of the disk's permanent address within the cluster and helps administrators identify which physical shelf a disk resides in during fault isolation.

Physical ID Assignment Process

  • 1Locate the small LCD display panel on the front or back of the shelf. Beside it you will find one or two physical buttons.
  • 2Press and hold the button. The display will cycle through numbers (0, 1, 2, 3 … up to 32).
  • 3Release when your desired shelf ID number appears. This immediately sets the shelf ID — no ONTAP command required.
  • 4Shelf IDs must be unique within the same SAS loop. Shelf IDs can be reused across separate loops (different SAS channels).
  • 5Valid range is 0–32 per loop, reflecting the maximum supported shelves per SAS channel on most controllers.
⚠️
Shelf ID Changes Are Disruptive in Production
Once a shelf is in service with active disk ownership and live data, changing its shelf ID is a disruptive operation that requires taking the shelf offline. Plan shelf IDs carefully before deployment. Shelf IDs can be freely changed only on shelves with no assigned disks.
✅ Allowed Same-Loop IDs
  • Each shelf must have a distinct ID within a single SAS loop
  • IDs in range 0–32 are valid (model-dependent upper limit)
  • Numbers need not be sequential — gaps are fine
✅ Cross-Loop ID Reuse
  • Two shelves in different SAS loops can share the same shelf ID number
  • Example: Loop A can have Shelf-ID 1 and Loop B can also have a Shelf-ID 1
  • ONTAP differentiates them by channel (loop) number

SECTION 08

Disk Ownership in ONTAP

In an HA pair, both controllers share visibility of all attached disks — every disk connected to the shelf stack can be seen by both nodes. However, each disk can only be owned by one controller at a time. Disk ownership determines which controller manages the disk and can use it in an aggregate.

Auto-Assign vs Manual Ownership

ONTAP can automatically assign disks to controllers based on physical shelf layout and an autoassignment policy, or you can assign disks manually using CLI commands. Disks that have not yet been assigned to any controller are in an unassigned state and appear in what is effectively a spare pool visible to both controllers.

Key Disk Ownership Commands

ONTAP CLI — Disk Ownership Operations
# View all disks and their ownership status ontap-cluster::> storage disk show Disk Owner Pool Status ---- ----- ---- ------ 1.1.0 node-a01 Pool0 Partner 1.1.1 node-a01 Pool0 Partner 1.1.23 - - Unassigned # View spare disks ontap-cluster::> storage aggregate show-spare-disks # Manually assign a disk to a specific node ontap-cluster::> storage disk assign -disk 1.1.23 -owner node-a01 # Remove disk ownership (make it unassigned) ontap-cluster::> storage disk removeowner -disk 1.1.23 # Enable auto-assign (on by default) ontap-cluster::> storage disk option modify -node node-a01 -autoassign on # Show disk ownership details per aggregate ontap-cluster::> storage aggregate show -fields node,raidtype,disks
ℹ️
Root Aggregate Disk Reservation
During initial cluster setup, ONTAP automatically reserves disks to create root aggregates — one per node. In non-ADP systems, exactly 3 full disks are consumed per controller (6 total for an HA pair). In ADP-enabled systems, only the root partition of multiple drives is used, leaving data partitions available for user aggregates.

SECTION 09

Advanced Disk Partitioning (ADP)

Advanced Disk Partitioning (ADP) is a storage efficiency technology that logically divides a physical drive into separate root and data partitions. Without ADP, ONTAP reserves entire disks for root aggregate use — which wastes significant capacity, especially on large-capacity or expensive flash drives.

The Problem ADP Solves

Consider a scenario with twelve 8TB NVMe SSDs in a shelf. Without ADP, ONTAP would reserve three entire disks (24TB) per controller simply for root aggregates — consuming 24TB of premium flash storage just for configuration data that realistically requires under 500GB. ADP eliminates this waste by partitioning each drive so only a small root partition is consumed by the root aggregate.

ADP Version 1 — FAS Mid-Range Systems

ADP v1 applies to FAS mid-range platforms and creates two partitions per drive: one root partition and one data partition. Key characteristics to understand:

  • ADP v1 is applied only to shelves included at initial cluster configuration — additional shelves purchased later will arrive as non-ADP by default
  • ADP and non-ADP disks cannot be mixed in the same aggregate — they must be placed in separate aggregates
  • Conversion from ADP to non-ADP (or vice versa) requires a complete cluster reset — it is destructive and not possible on a system with live production data
  • Optional for larger FAS 8000/9000 platforms; factory-configured for FAS 2500, 2600, 2700 series

ADP Version 2 — AFF All-Flash Systems

ADP v2 is mandatory on all AFF systems. It creates three partitions per drive: one root partition and two equal data partitions. This further maximizes flash utilization and ensures even data distribution across mirrored configurations.

  • Mandatory on all AFF platforms — cannot be disabled
  • Three-partition model: 1 root + 2 data partitions per drive
  • Data partitions are equal in size — ONTAP divides remaining capacity in half
  • All shelves (including future expansions) are automatically ADP v2 on AFF systems
  • Enables flexible aggregate design — data partitions can be assigned independently

Verifying ADP Configuration

ONTAP CLI — ADP Verification Commands
# Enter Diagnostic mode (required for partition commands) ontap-cluster::> set diag ontap-cluster::*> # Show disk partitions — if output is populated, ADP is active ontap-cluster::*> disk partition show Disk Part Type Owner Pool Size -------- ---- ---- ----- ---- ---- 1.1.0P1 P1 root node-a01 Pool0 400.0GB 1.1.0P2 P2 data node-a01 Pool0 3.58TB 1.1.0P3 P3 data node-a01 Pool0 3.58TB # 3 partitions per disk = ADP v2 (AFF) # 2 partitions per disk = ADP v1 (FAS) # Empty output = No ADP (full disk, non-partitioned) # Check usable capacity breakdown (data vs root) ontap-cluster::*> storage aggregate show-spare-disks local data usable: 3.58TB local root usable: 400.0GB # Return to Admin mode after diagnostics ontap-cluster::*> set admin
💡
Identifying ADP Version from Output
If disk partition show returns 3 partitions per disk (P1, P2, P3), you have ADP v2. Two partitions per disk indicates ADP v1. Empty output confirms the cluster is operating in non-ADP mode with full physical disks.

SECTION 10

Shelf & Disk Firmware Management

Disk shelves and the drives inside them each carry their own firmware. Keeping firmware current ensures compatibility, stability, and access to bug fixes. Understanding how firmware management works in modern ONTAP versions will save you significant operational overhead.

🕒 Legacy Approach (Pre-ONTAP 9)
  • ONTAP OS upgrade — separate manual process
  • Shelf firmware upgrade — separate manual process
  • Disk firmware upgrade — separate manual process
  • Each required individual planning, scheduling, and validation
✅ Modern Approach (ONTAP 9+)
  • Shelf firmware, disk firmware bundled inside ONTAP image
  • All components upgraded automatically during ONTAP upgrade
  • Background non-disruptive updates during normal operation
  • Individual firmware update only needed for specific patch fixes
ℹ️
When Individual Firmware Updates Are Still Needed
If after an ONTAP upgrade you discover a firmware bug in a specific shelf or disk model, the vendor may release an out-of-cycle firmware patch. In this case, you apply only that specific firmware package independently. This is now the exception rather than the standard practice.

SECTION 11

Scenario-Based Interview Q&A

Q1 You're in the ONTAP CLI and need to run a command that isn't available in admin mode. What do you do, and what precaution do you take?
Enter advanced mode using set adv. The prompt will display a * confirming elevated privileges. After completing the task, return to admin mode with set adm. The precaution is to note which mode you are in before running any command — the asterisk in the prompt is your visual cue. Never leave an interactive session running in advanced or diagnostic mode unattended.
Q2 Your team purchased 3 additional SAS shelves to expand storage. Two are DS224C (IOM12) and one is a DS2246 (IOM6). How do you connect them?
The DS224C shelves (IOM12, 12Gb/s) and the DS2246 shelf (IOM6, 6Gb/s) must not be daisy-chained together — mixing IOM types on a single loop degrades performance. Connect the two DS224C shelves into one new SAS loop and the DS2246 into a separate loop, each using its own pair of SAS ports on the controller (or via a SAS HBA expansion card if native ports are exhausted).
Q3 A disk alert shows failed disk 2.3.15. How do you physically locate it?
Decode the disk ID: Channel 2 (the second SAS loop group), Shelf ID 3 (the shelf with ID set to "3" on that loop), disk slot 15 (16th physical slot, counting from 0). Walk to the shelf rack, identify channel 2's cable grouping, find the shelf with "3" on its LCD, and count to slot 15. The disk LED should also be blinking red to confirm the location.
Q4 What is ADP and why is it mandatory on AFF systems?
ADP (Advanced Disk Partitioning) logically splits physical drives into root and data partitions so the root aggregate only consumes a small partition (~400GB) rather than entire drives. On AFF systems — which use expensive NVMe SSDs — wasting multiple full drives just for root configuration would be financially and operationally unacceptable. ADP v2 (the AFF version) creates three partitions per drive (1 root + 2 data), ensuring near-zero overhead from root aggregates regardless of drive capacity.
Q5 A client asks if they can mix ADP and non-ADP drives in the same data aggregate after expanding storage. What is your answer?
No — ADP and non-ADP disks cannot coexist in the same aggregate. They must be placed in separate aggregates. This means the client will have two distinct capacity pools. If they want uniformity, the only option is to either request that expansion shelves be configured as ADP at the factory, or reset the entire array to convert — which is disruptive and not possible on a system with live data.
Q6 How do you verify whether a running ONTAP cluster is using ADP, and which version?
Enter diagnostic mode (set diag) and run disk partition show. If output is empty, no ADP is in use. If you see 2 partitions per disk (P1, P2), it is ADP v1 (FAS mid-range). If you see 3 partitions per disk (P1, P2, P3), it is ADP v2 (AFF). You can also cross-reference partition sizes — the root partition will be significantly smaller than the data partition(s).

SECTION 12

Best Practices Summary

🖥️ CLI Mode Safety
  • Always use set <TAB> to confirm your current privilege level before running critical commands
  • Return to admin mode immediately after completing advanced tasks
  • Never execute unfamiliar commands in diagnostic mode without vendor guidance
  • Document any diagnostic-mode actions taken during troubleshooting
🔗 Shelf Connectivity
  • Never mix different IOM generations (IOM3, IOM6, IOM12) in a single SAS loop
  • Always use separate loops for different shelf models
  • Add/remove shelves one path at a time for non-disruptive expansion
  • Plan and cable both IOM-A and IOM-B paths on every shelf for full redundancy
🏷️ Shelf & Disk IDs
  • Plan all shelf IDs before racking — avoid changes after go-live
  • Keep a physical cable and shelf ID map document for each environment
  • Use consistent ID numbering conventions across deployments
  • Enable disk autoassignment in new environments to reduce manual overhead
✂️ ADP Planning
  • Confirm ADP requirements before placing hardware orders — factory configuration is easiest
  • Never mix ADP and non-ADP disks in the same aggregate
  • Plan aggregate layout carefully when using ADP v1 with mixed shelf purchases
  • For AFF — rely on ADP v2 default behavior; do not attempt to disable it
