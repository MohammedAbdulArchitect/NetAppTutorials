SECTION 01

How Snapshots Work

An ONTAP snapshot is a point-in-time, read-only image of a volume. Unlike traditional backup copies that duplicate all data, a snapshot stores only metadata pointers to existing blocks at the moment of creation — making it near-instantaneous and initially space-free.

Snapshots can be created at volume level and aggregate level. However, NetApp strongly recommends volume-level snapshots only (covered in Section 02).

Copy-on-Write: What Happens After Snapshot Creation

Once a snapshot exists, ONTAP implements Copy-on-Write (CoW): before any live block is modified or deleted, its original content is preserved inside the snapshot's reserved space. The snapshot grows only as data changes accumulate.

After modifications and deletions — snapshot grows to hold changed/deleted blocks; unchanged blocks share pointers
🧠
Key Insight — Data Protection Window
A snapshot only protects data that existed at the time of its creation. Any data written and then deleted between two consecutive snapshot checkpoints cannot be recovered from either snapshot. The protection window is the interval between snapshots — shorter intervals mean smaller gaps in recoverability.

SECTION 02

Snapshot Reserve Space

ONTAP carves out a dedicated portion of each volume (and historically, aggregate) as snapshot reserve — space exclusively for snapshot blocks. Understanding how this reserve changed between ONTAP versions is a common interview topic.

⚠️ ONTAP ≤ 7.3 (Legacy)
Volume snapshot reserve:20%
Aggregate snapshot reserve:5%
On a 100 GB volume, only 80 GB was usable for data
✅ ONTAP 8.0+ (Current)
Volume snapshot reserve:5%
Aggregate snapshot reserve:0%
Aggregate-level snapshots are no longer recommended

Why Aggregate Snapshots Are Discouraged

  • Restoring an aggregate snapshot would affect all volumes on that aggregate simultaneously — collateral impact is unacceptable in most cases
  • Aggregate-level snapshots introduce measurable I/O performance overhead due to the scale of tracked block changes
  • Granular recovery is not possible — you cannot restore a single file or single volume from an aggregate snapshot
⚠️
Best Practice
Always operate with volume-level snapshots only. Set aggregate snapshot reserve to 0 (it already defaults to 0 from ONTAP 8.0+). If you encounter an older system still using aggregate snapshots, evaluate migrating to a volume-level strategy.

Setting Snapshot Reserve at Volume Creation

ONTAP CLI
# Create volume with custom snapshot reserve ::> volume create -vserver svm-prod01 -volume vol_appdata \ -aggregate aggr1_node01 -size 200g \ -junction-path /vol_appdata \ -percent-snapshot-space 10 # 10% = 20 GB reserved for snapshots; 180 GB usable for data # Verify snapshot reserve on all volumes ::> volume show -fields percent-snapshot-space

SECTION 03

Creating & Viewing Snapshots

Snapshots can be created on-demand (manually) or automatically via a schedule. Both methods use the same underlying volume snapshot command namespace.

ONTAP CLI — Snapshot Lifecycle
# Create a manual snapshot ::> volume snapshot create -vserver svm-prod01 -volume vol_appdata \ -snapshot snap_pre_patching_20240515 # List all snapshots on a volume ::> volume snapshot show -vserver svm-prod01 -volume vol_appdata ---Blocks--- Vserver Volume Snapshot State Size Total %Inode -------- -------- ------------------------ ----- ---- ----- ------ svm-prod01 vol_appdata snap_pre_patching_20240515 valid 48KB 10% 0% # Delete a specific snapshot (admin only) ::> volume snapshot delete -vserver svm-prod01 -volume vol_appdata \ -snapshot snap_pre_patching_20240515
💡
Initial Snapshot Size
A newly created snapshot consumes only a few KB (typically 4–20 KB). It grows only as data on the volume changes. After heavy write activity, a snapshot may grow to several GB as it accumulates block deltas.

SECTION 04

Restoring Data from Snapshots

ONTAP supports two distinct restore approaches. Choosing the right method prevents unnecessary disruption to other data on the volume.

🔁 Volume-Level Restore

Reverts the entire volume to the state at snapshot creation time. All data written after that snapshot is permanently discarded.

Use with caution CLI command
📄 File / Folder Restore

Navigate the hidden .snapshot directory and manually copy only the needed files back. Selective and non-disruptive.

Preferred for granular recovery

Volume-Level Restore (CLI)

ONTAP CLI — Full Volume Restore
# Restore entire volume to a specific snapshot ::> volume snapshot restore -vserver svm-prod01 \ -volume vol_appdata \ -snapshot snap_pre_patching_20240515 # ⚠️ This is irreversible — all post-snapshot data is lost
🚫
Warning — Volume Restore Is Destructive
Performing a volume-level snapshot restore permanently discards all data written after the chosen snapshot. Always confirm with the application team before proceeding. For single-file or folder recovery, use the .snapshot directory method instead.

Granular File Restore via NFS Client

On NFS-mounted volumes, ONTAP exposes a hidden .snapshot directory at the root of every mount point. Clients can browse this directory to access any historical snapshot and copy files directly — no admin intervention required.

Linux Shell — NFS Snapshot Browse & Restore
# List all contents including hidden items root@linux-host:~# ls -a /mnt/vol_appdata/ . .. app_configs logs .snapshot # Browse available snapshots root@linux-host:~# ls /mnt/vol_appdata/.snapshot/ snap_pre_patching_20240515 hourly.2024-05-15_1005 # Access files inside a specific snapshot root@linux-host:~# ls /mnt/vol_appdata/.snapshot/snap_pre_patching_20240515/ app_configs logs settings.conf run.sh # Copy a specific file back to live location root@linux-host:~# cp /mnt/vol_appdata/.snapshot/snap_pre_patching_20240515/settings.conf \ /mnt/vol_appdata/settings.conf
✅
NFS vs CIFS Snapshot Visibility
On NFS, the .snapshot directory is visible by default. To hide it, disable the option with vserver nfs modify -vserver <name> -v3-hide-snapshot enabled. On CIFS/SMB, snapshots are hidden by default. They become visible when the Show Snapshot share property is enabled per-share.

Snapshot Immutability

Snapshot data is immutable — end users and server teams cannot modify or delete files inside the .snapshot directory, even if they have full write access to the live volume. Any attempt to delete or overwrite snapshot content returns a permission error.

flowchart TD A[User or Application] -->|"Try to DELETE\nfile in .snapshot"| B{ONTAP Kernel\nPermission Check} B -->|"Snapshot = Immutable"| C["❌ Operation Denied\n'Read-only file system'"] B -->|"Live volume data"| D["✅ Operation Allowed"] A -->|"Try to MODIFY\nfile in .snapshot"| B style A fill:#1c2333,stroke:#58a6ff,color:#e6edf3 style B fill:#1c2333,stroke:#e3b341,color:#e6edf3 style C fill:#1c2333,stroke:#f78166,color:#f78166 style D fill:#1c2333,stroke:#3fb950,color:#3fb950
Snapshot immutability is enforced at the ONTAP kernel level — not configurable by end users
📝
Admin vs End-User Permissions
While end users cannot delete snapshot data, a cluster admin can delete an entire snapshot using volume snapshot delete. This removes the snapshot and frees any space it was consuming. SnapLock is a separate, licensed feature that prevents even admins from deleting snapshot data — covered in advanced topics.

SECTION 05

CRON-Based Snapshot Schedules

ONTAP uses a CRON-style job scheduler to define when snapshots are created. Schedules are decoupled from snapshot policies, allowing the same schedule to be reused across multiple policies.

Default Schedules Available

5min
Every 5 min
hourly
HH:05 each hour
daily
00:10 daily
weekly
Sunday 00:15
monthly
1st of month
8hour
02:15, 10:15, 18:15

Creating a Custom Schedule

ONTAP CLI — Custom CRON Schedule
# View existing schedules ::> job schedule cron show # Create a schedule: every Tuesday & Friday at 03:00 AM ::> job schedule cron create \ -name tue_fri_03am \ -minute 0 \ -hour 3 \ -dayofweek Tuesday,Friday # Create a schedule: 1st of every month at 10:30 AM ::> job schedule cron create \ -name monthly_1st_1030 \ -minute 30 \ -hour 10 \ -day 1 # Note: -day and -dayofweek cannot be used simultaneously # Verify created schedule ::> job schedule cron show -name tue_fri_03am
💡
Cluster-Level vs SVM-Level Schedules
Schedules created at the cluster level are reusable across all SVMs. Schedules created within a specific SVM context are scoped to that SVM only. Best practice is to create shared schedules at the cluster level for maximum reusability.

SECTION 06

Snapshot Policies

A snapshot policy bundles up to five schedules with their retention counts (how many snapshots of each type to keep) and optional naming prefixes. Applying a policy to a volume automates the entire snapshot lifecycle.

flowchart LR SCH1["⏰ Schedule: hourly\nRetain: 12"] SCH2["⏰ Schedule: daily\nRetain: 7"] SCH3["⏰ Schedule: weekly\nRetain: 4"] SCH4["⏰ Schedule: monthly\nRetain: 12"] SCH5["⏰ Custom Schedule\nRetain: 6"] POL["📋 Snapshot Policy\nvol-prod-snaps"] VOL["📦 Volume\nvol_appdata"] SCH1 --> POL SCH2 --> POL SCH3 --> POL SCH4 --> POL SCH5 --> POL POL -->|"Applied to"| VOL style POL fill:#1c2333,stroke:#bc8cff,color:#e6edf3 style VOL fill:#1c2333,stroke:#58a6ff,color:#e6edf3
A snapshot policy can combine up to 5 schedules — each with its own retention count — applied to one or many volumes
ONTAP CLI — Snapshot Policy Creation
# View existing policies ::> volume snapshot policy show # Create a multi-schedule snapshot policy ::> volume snapshot policy create \ -policy vol-prod-snaps \ -enabled true \ -schedule1 hourly -count1 12 -prefix1 hrly \ -schedule2 daily -count2 7 \ -schedule3 weekly -count3 4 \ -schedule4 monthly_1st_1030 -count4 12 \ -schedule5 tue_fri_03am -count5 8 # Apply policy to a volume ::> volume modify -vserver svm-prod01 -volume vol_appdata \ -snapshot-policy vol-prod-snaps # Apply "none" policy to disable automated snapshots ::> volume modify -vserver svm-prod01 -volume vol_appdata \ -snapshot-policy none

Retention Count Behaviour

The -count parameter controls how many snapshots created by that schedule are kept simultaneously. When the count is exceeded, the oldest snapshot from that schedule is automatically deleted — a rolling window mechanism.

When count=4 and the 5th snapshot is created, the oldest is automatically purged — maintaining exactly 4

Snapshot Naming with Prefix

By default, auto-created snapshots use the schedule name as a prefix (e.g., hourly.2024-05-15_1005). A custom prefix can be set per schedule entry in the policy. This helps with filtering and autodelete prefix matching.

Volume Autosize

Volume Autosize is an ONTAP feature that automatically grows (and optionally shrinks) a volume's allocated size based on usage thresholds — eliminating the need for constant manual capacity monitoring.

How Autosize Works

Volume autosize loop — ONTAP monitors usage and triggers growth when threshold is crossed
ONTAP CLI — Configure Volume Autosize
# Check current autosize status ::> volume autosize -vserver svm-prod01 -volume vol_appdata # Enable autosize in GROW mode ::> volume autosize -vserver svm-prod01 -volume vol_appdata \ -maximum-size 300g \ -grow-threshold-percent 98 \ -mode grow # Current: 200 GB → will grow up to 300 GB when usage hits 98% # Verify configuration ::> volume autosize -vserver svm-prod01 -volume vol_appdata Volume vol_appdata is set to grow to a maximum of 300GB when volume used space is above 98%. Mode: grow.

Autosize Modes

🔴
Off
Autosize disabled. Volume must be manually expanded when full.
📈
Grow
Volume grows automatically when threshold is hit, up to the configured maximum. Volume never shrinks.
↕️
Grow + Shrink
Volume grows on high usage AND shrinks when usage drops below the shrink threshold, down to a configured minimum size.

Grow + Shrink Configuration

ONTAP CLI — Grow + Shrink Mode
::> volume autosize -vserver svm-prod01 -volume vol_appdata \ -maximum-size 300g \ -minimum-size 200g \ -grow-threshold-percent 98 \ -shrink-threshold-percent60 \ -mode grow_shrink # Shrinks when used space drops below 60%; never goes below 200 GB minimum
🧠
Grow + Shrink Caution
While grow+shrink optimises aggregate space efficiency, be careful with the minimum size and shrink threshold. ONTAP will not shrink a volume below the amount of data it contains. Set a realistic minimum that accounts for typical data patterns. Most production environments use grow-only mode for predictability.

SECTION 08

Snapshot Autodelete

Snapshot Autodelete is a safety net that automatically removes snapshots when a volume is approaching capacity — preventing volumes from going full due to snapshot accumulation. This is distinct from policy-based retention.

flowchart TD A[Volume fills up — snapshots accumulating] --> B{Trigger threshold\nreached?} B -->|"No"| A B -->|"Yes"| C[Autodelete activates] C --> D{Commitment\nMode?} D -->|"try"| E["Delete only free\n(unlocked) snapshots ✅"] D -->|"disrupt"| F["Delete replication snapshots\n⚠️ May break SnapMirror"] D -->|"destroy"| G["Delete 3rd-party\nbackup snapshots ⚠️"] E --> H{Target free space\nachieved?} H -->|"No"| E H -->|"Yes"| I[Autodelete stops] style A fill:#1c2333,stroke:#58a6ff,color:#e6edf3 style B fill:#1c2333,stroke:#e3b341,color:#e6edf3 style C fill:#1c2333,stroke:#39d3c3,color:#e6edf3 style D fill:#1c2333,stroke:#bc8cff,color:#e6edf3 style E fill:#1c2333,stroke:#3fb950,color:#3fb950 style F fill:#1c2333,stroke:#e3b341,color:#e3b341 style G fill:#1c2333,stroke:#f78166,color:#f78166 style H fill:#1c2333,stroke:#39d3c3,color:#e6edf3 style I fill:#1c2333,stroke:#3fb950,color:#e6edf3
Autodelete flow — the commitment mode determines which snapshots are eligible for deletion

Commitment Mode Options

try ✅
Deletes only free (unlocked) snapshots. Does not impact SnapMirror, SnapVault, or backup application snapshots. Recommended for all environments.
disrupt ⚠️
Also deletes snapshots used by replication (SnapMirror). Will break active replication relationships. Use only if you understand the impact.
destroy ⛔
Deletes snapshots held by third-party backup tools. May corrupt ongoing backup jobs. Rarely justified in production.

Deferred Delete Options

OptionEffect
scheduledDeletes only auto-created (scheduled) snapshots first
user_createdDeletes only manually created snapshots first
prefixDeletes snapshots whose names start with a specified prefix
noneNo preference — deletes oldest snapshots regardless of origin
ONTAP CLI — Configure Snapshot Autodelete
# Enable snapshot autodelete on a volume ::> volume snapshot autodelete modify \ -vserver svm-prod01 \ -volume vol_appdata \ -enabled true \ -trigger volume \ -target-free-space15 \ -delete-order oldest_first \ -commitment try \ -defer-delete scheduled # Triggers when volume is full; stops when 15% free space is restored # Deletes oldest scheduled snapshots first; never touches locked snapshots # Verify configuration ::> volume snapshot autodelete show \ -vserver svm-prod01 -volume vol_appdata
✅
Best Practice — Autodelete Settings
Always use commitment: try. Set the trigger to volume (not snapshot-reserve, since that should be 0 or low). Target free space of 10–20% gives breathing room. Use deferred-delete: scheduled so manually created important snapshots are preserved while routine scheduled ones are cleaned first.

SECTION 09

Space Management Priority

When both Volume Autosize and Snapshot Autodelete are configured on the same volume, ONTAP allows you to control which mechanism runs first when the volume fills up. This is configured via the space-mgmt-try-first parameter.

Priority Options

Option A — Default
Volume Grow First
→
Fallback
Snap Autodelete
Option B
Snap Autodelete First
→
Fallback
Volume Grow
ONTAP CLI — Set Space Management Priority
# Set volume grow as first responder (default and recommended) ::> volume modify -vserver svm-prod01 -volume vol_appdata \ -space-mgmt-try-first volume_grow # Set snap autodelete as first responder ::> volume modify -vserver svm-prod01 -volume vol_appdata \ -space-mgmt-try-first snap_delete
🧠
Why Volume Grow First Is the Default
Deleting snapshots means compromising data protection history. If aggregate space allows, growing the volume first preserves your snapshot archive intact. Autodelete should be the last resort. The default setting reflects this safety-first philosophy — always exhaust the grow option before sacrificing snapshots.

SECTION 11

Scenario-Based Interview Q&A

Q1 Explain how Copy-on-Write works in ONTAP snapshots and why snapshots start with almost zero size.
At the moment of snapshot creation, ONTAP stores only lightweight metadata pointers to all existing live blocks — no data is duplicated. The snapshot consumes just a few KB. As data on the live volume is modified or deleted, the original blocks are redirected into the snapshot's space before being overwritten. This means the snapshot grows incrementally only as changes accumulate. If no data changes occur after snapshot creation, the snapshot remains near-zero in size indefinitely.
Q2 A user accidentally deleted a critical configuration file on an NFS share. The admin does not want to do a full volume restore. What is the procedure?
Navigate to the root of the NFS mount point and run ls -a to reveal the hidden .snapshot directory. Enter that directory and list the available snapshots by name and timestamp. Identify the snapshot taken before the deletion occurred, browse into it, locate the deleted file, and use cp to copy it back to its original path. This is non-disruptive — no volume restore is needed and no other data is affected.
Q3 Why does NetApp no longer recommend aggregate-level snapshots, and what is the default reserve set to?
From ONTAP 8.0 onwards, NetApp changed the default aggregate snapshot reserve to 0% and deprecated the practice. Two reasons drive this: (1) Restoring an aggregate snapshot impacts all volumes on that aggregate simultaneously — there is no way to restore a single volume or file without affecting neighbours. (2) Aggregate-level snapshots introduce significant performance overhead due to the volume of block-change tracking required at that scale. Volume-level snapshots provide granular, targeted protection without these drawbacks.
Q4 What is the difference between Volume Autosize and Snapshot Autodelete, and when should each trigger?
Volume Autosize grows (and optionally shrinks) the volume's allocated size from the aggregate — adding more capacity without admin intervention. Snapshot Autodelete removes existing snapshots to reclaim space already allocated to the volume. They are complementary: Autosize extends the volume's ceiling; Autodelete cleans up internal space consumers. The space-mgmt-try-first parameter controls which runs first when the volume fills. The recommended approach is to attempt volume growth first (preserving snapshot history) and only resort to autodelete as a fallback.
Q5 What happens if a volume has snap autodelete enabled with commitment=disrupt?
The disrupt mode allows ONTAP to delete snapshots that are currently in use by SnapMirror replication. If a SnapMirror baseline or delta snapshot is deleted, the replication relationship will fail and may require a full re-baseline. This is why try is strongly recommended in all production environments — it only deletes free (unlocked) snapshots that are not referenced by any ONTAP service, leaving replication and backup snapshots intact.
Q6 Can an end user delete snapshot data they can see in the .snapshot directory?
No. Snapshots are immutable — ONTAP enforces this at the kernel level. Even if a user has full read/write permissions on the live volume, any attempt to modify, delete, or overwrite files inside the .snapshot directory returns a "read-only file system" or "permission denied" error. Only a cluster administrator can delete an entire snapshot via volume snapshot delete, which removes the snapshot object — not individual files within it.
Q7 How does snapshot retention work when count is set to 12 for the hourly schedule?
ONTAP maintains a rolling window of exactly 12 hourly snapshots. When the 13th hourly snapshot is created, the oldest hourly snapshot is automatically purged to bring the count back to 12. This cycle continues indefinitely. The retention count applies per schedule within a policy — so a policy with both hourly (count 12) and daily (count 7) schedules maintains up to 12 hourly + 7 daily = 19 snapshots simultaneously on the volume.
